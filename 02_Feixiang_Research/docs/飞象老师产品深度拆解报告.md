## 执行摘要

飞象老师是一个**面向 K12 一线教师的 AI 教学内容生成平台**：教师用一句话（或上传一份资料）描述教学需求，平台经多阶段 agent / workflow 编排，产出**自包含、可交互的 HTML 课件 / 试题 / 教案**。产品代号 Musk AI Studio，归属猿辅导系基础设施（`fbcontent.cn` / `zhenguanyu` / `octopus`）。

**最值得团队关注的 5 个判断**：

1. **飞象是"开环生成器"，这是战略选择，不是产品缺陷。** 全站埋点只采集页面停留时长，一字未采学生作答数据，无 SCORM / xAPI，与任何 LMS / 班级 / 课堂零集成。一个连停留秒数都做 100ms 防抖精细累加的团队，不可能"忘了"采作答——这说明它主动押注"覆盖最广、集成成本最低的开环赛道"。这一选择带来了规模与速度，代价是永远拿不到数据闭环这条护城河。

2. **内容生成架构是"外壳工程化 + 内层软规范"，这个二元结构决定了它的全部能力边界。** 跨 4 个课件样本验证：共享库、iframe 宿主、埋点、后处理噪声层跨样本一致（硬资产）；而设计令牌取值、DOM、CSS、交互函数逐课件各异（软规范）。这一架构让它"能演示任意互动"，却"天然采不到结构化作答数据"——不是有人没想到，而是架构在起点就分叉了。

3. **生成链路工程化程度高，值得参考。** 8+ 阶段编排、2 个 HITL 确认门控、复杂度感知路由、交互剧本中间层、9 Batch 串行 + 骨架注入、增量定向修复——整条生成侧流水线已验证可行，是我方"可抄作业"的部分。

4. **短轮询替代 SSE 是一个经过权衡的工程选择，不是能力不足。** agent runner 以约 80–150ms/次轮询状态端点。在分钟级长任务、多阶段 workflow 场景下，SSE 的长连接风险（网关超时、重连逻辑复杂）比短轮询的高请求率代价更难控制。这个选择在当前规模合理，但在高并发场景会是瓶颈。

5. **资源广场 + 改编机制是飞象的增长飞轮，PGC 冷启动 + UGC 复利是它的社区核心逻辑。** TeacherYO 官方账号的 PGC 示范内容（6万+浏览量）解决了冷启动，历史任务里大量"改编《…》"证实了改编是最高频的真实用法，"看→改→再生产"的正循环是飞象最难被单点复制的竞争壁垒。

> **对「ClassIn AI 助教升级」项目的战略启示**：飞象的终点（下载一个独立 HTML）恰是 ClassIn 的起点。我方的差异化不在"生成得比飞象更快或更好看"，而在**把生成内容投进真实教学场景（预习/课堂/作业）并把作答数据收回来反哺生成**。要做到这一点，有一道飞象没有、我方必须在 Phase 0 就立好的地基：**内容 schema / DSL + 题型组件 + 运行时 player**。这不是技术洁癖，而是能否做数据闭环的底层前提——详见结尾对照章节。

# 第一部分 · 产品侧

## 1. 产品定位与目标用户

飞象老师把自己钉在一个非常具体的位置：**教师的 AI 备课工作台**。它不卖"AI 聊天"，也不卖"通用文档工具"，而是卖"做一节课所需的全套内容产物"。

| 维度 | 内容 | 证据 |
|---|---|---|
| 目标用户 | K12（中小学）一线教师，覆盖小学一年级到高中 | 年级筛选器含"一年级…九年级、高中" |
| 核心场景 | 备课提效：互动课件、试题命制、组卷、大单元教案 | 4 个 AI 工具 + 资源广场 |
| 核心价值 | 用 AI 把"做一节课"自动化：课标研读→教学设计→逐页生成→可交互产物 | |
| 计费 | 积分制（实勘账号底部显示"剩余积分：54"） | |
| 社区氛围 | 资源广场 UGC/PGC 混合：作者署名、浏览量、热门/新手角标 | |

### 1.1 它押注了什么

飞象赌的是一个体验落差：教师备课是"高耗时、高重复、有专业门槛"的日常工作，而 AI 生成的冲击感在这个场景里远大于"润色一段文字"——因为产物是"一节完整的可交互课件"，不是文本片段。"一句话出 20 页带动画的课件"对教师的震撼感，就是飞象的核心获客逻辑。

**这个赌注的逻辑链**：教师备课耗时 → AI 能大幅压缩 → 落差足够大 → 愿意付积分 → 愿意推荐同事 → 资源广场内容沉淀 → 更多教师来看 → 形成飞轮。这条链目前看来是跑通的（实勘历史任务 27+ 条、广场内容 6 万+浏览量是真实使用的证明）。

### 1.2 积分制的商业化逻辑

积分制是飞象把"生成一次"变成**可计量价值单元**的手段。这个设计有几层含义值得注意：

**计量单位选择的合理性**：积分与"生成次数"挂钩，而不是"功能权限"或"时间订阅"。这是因为 AI 生成课件的成本高度非线性——一次 20 页复杂互动课件的 LLM 调用成本，可能是一句命题的 10 倍。按次计费（积分）比按月订阅更公平，也更能反映用户实际使用价值。

**积分制对用户行为的影响**：当用户知道"生一次消耗积分"，他们会自然而然地更精心填写需求、更少乱试。这客观上提升了有效任务比例，降低了低质量生成对服务器的压力。

**商业化阶段判断**：从实勘看，飞象已进入商业化阶段（积分有限、积分可充值推断），这意味着产品已过"免费拉新"阶段，在测试用户为优质内容付费的意愿。这对 ClassIn 的产品规划有参考意义——AI 生成内容的用户付费转化，飞象已在跑路径。

## 2. 全站功能地图

飞象的功能由三块拼成：**4 个 AI 创作工具 + 任务管理系统 + 资源广场**。三者通过左侧栏统一组织，靠 Hash 路由的 `featureId` 参数切换。

```
飞象老师（feixianglaoshi.com，#/home Hash 路由）
│
├── ① 核心 AI 创作工具组（左侧栏顶部，featureId 切换）
│   ├── AI互动课件    #/home?featureId=20  ← 资源广场默认页
│   ├── AI命题        #/home?featureId=17
│   ├── AI组题        #/home?featureId=18
│   └── AI教案·大单元 #/home?featureId=19
│       （17/18/19 在对话页顶部以 Tab 互相切换；20 为独立的广场页）
│
├── ② 任务管理系统（左侧栏）
│   ├── + 新建任务（绿色 CTA，置顶）
│   ├── 置顶任务（暴露 5 条"能力线索"任务）
│   ├── 今日任务
│   └── 历史任务（可滚动，实测 27+ 条：
│       "改编《3D球形学生随机点名程序》"
│       "生成帕斯卡原理互动教学动画"
│       "根据文章命制北京中考英语阅读题"…）
│
├── ③ 资源广场（AI互动课件页主内容区）
│   ├── 顶部 Slogan + 多模态输入框 + 4 个能力按钮
│   │   （教学动画 / 教学游戏 / 互动课件 / 数据回收）
│   ├── 学科分类导航（横向 Tab，12 类，数学带"热"角标）
│   ├── 二级筛选器：教材版本 × 年级(一~九/高中) × 学期(上/下册)
│   └── 资源卡片瀑布流（每行 4 列：封面+标题+作者+浏览量+角标）
│
└── ④ 核心创作流（对话页，提交需求后进入）
    信息确认表单 → 专家思考折叠卡片 → 课件生成计划 →
    分批生成进度 → 交付物卡片 → 追问修复闭环
```

### 2.1 置顶任务暴露的两种生成范式

置顶任务的命名直接暴露了飞象内部的生成机制分层：

| 置顶任务名 | 范式 | 含义 |
|---|---|---|
| 教学动画 - **Agent自主规划** | Agent 范式 | Agent 动态决定步骤 |
| 练习数据回收 - **Agent自主调用** | Agent 范式 | Agent 自主调用工具 |
| 互动游戏 - **Agent自主调用** | Agent 范式 | Agent 自主调用工具 |
| 文档生成 - **工作流生成** | Workflow 范式 | 固定流水线，可控 |
| 多页互动课件 - **工作流生成** | Workflow 范式 | 固定流水线，可控 |

**判断**：飞象没有把所有任务都压进一种范式，而是让复杂度决定路由——多页课件这类"高成本、高质量要求"的任务走可控 Workflow，灵活任务走 Agent。这是工程成熟度的表现，也是我方可直接参考的编排策略。

### 2.2 四个 AI 工具的"一个壳、多个能力"设计

17/18/19/20 四个工具共享同一个对话壳与侧栏结构，差异仅在示例提示词与后端 Skill 配置上。**这不是偶然**——这是一个明确的架构决策：让新增工具的前端成本趋近于零（只要新增 featureId + Skill + 示例提示词，即可对外呈现为独立工具）。飞象能快速横向扩展命题/组题/教案/课件四个品类，工程基础就在这里。

## 3. 核心增长飞轮：资源广场与改编机制

这是飞象产品策略中最值得深入拆解的部分，也是当前报告最容易被忽视的增长逻辑。

### 3.1 增长飞轮的运作机制

飞象的内容飞轮由三个环节构成：

```
PGC 示范内容（TeacherYO 官方账号）
        ↓
教师浏览资源广场 → 看到高质量示范课件 → 降低"不知道 AI 能做成什么"的认知门槛
        ↓
"改编"按钮触发 → 一键复用他人作品骨架 → 用最低的输入成本生成自己的版本
        ↓
教师生产的 UGC 内容 → 上传广场 → 供更多人浏览、改编
        ↓
广场内容密度增加 → 更多教师来逛 → 更多人生成 → 回到起点
```

**关键设计：PGC 解决冷启动，UGC 产生复利。**

冷启动阶段，飞象用 TeacherYO 官方账号密集发布高质量 PGC 内容（"新手使用攻略"、"开放日直播"、"必看小技巧"、各学科示范课件），给广场填充了足够的参考样本。实勘数据：TeacherYO 账号作品浏览量普遍 6000+，最高 6.5 万——这是典型的"锚点内容"，既教会教师平台能做什么，又制造了改编的起点。

**改编作为一等公民的增长含义**：历史任务列表里，"改编《…》"出现的频率与"生成…"相当。这不是用户偶然行为，而是飞象刻意设计的——改编的门槛比从零生成低得多（教师只需确认一份已有骨架的参数，而不用凭空描述需求），成功率更高，教师更不容易"不知道怎么开口"。

**这对飞象增长意味着什么**：改编机制把每一份 UGC 课件都变成了潜在的病毒载体——被改编一次，相当于推荐了一次。广场里内容越多，改编行为越多；改编越多，UGC 越多；UGC 越多，广场越丰富。这是一个正向的内容复利循环，也是飞象最难被单点复制的竞争壁垒（因为内容沉淀需要时间积累）。

### 3.2 "改编优先"的反向含义

如果一个产品把"改编他人作品"设计为一等公民，就意味着它默认用户不擅长从零输入需求。这是一个重要的**用户洞察**：教师面对空白输入框的焦虑感很强，有一个"起点"就能大大降低放弃率。

飞象对这个洞察的回应不只是"改编"按钮，还包括：示例提示词（"大家都在用"列表）+ 热度数字（营造从众）+ AI 预填参数表单（默认值降低决策成本）。每一层都是在降低"第一步"的摩擦。

**对 ClassIn 的启示**：如果 ClassIn 做内容生成，一定要考虑"起点"设计——是用教师自己的历史备课资料？是学科模板？是已有题库？"从什么出发"比"生成什么"更影响首次使用体验。

### 3.3 资源详情页把"浏览"变成了"转化漏斗"

这次继续实勘后，飞象的增长设计比原先看到的更完整。资源广场不是停留在"卡片列表"，而是有一套明确的**详情页转化结构**：

```
广场卡片
  ↓
资源详情页（大预览 iframe）
  ↓
作者信息 + 关注 + 分享 + 一键改编 + 下载
  ↓
同题改编链 + 泛相关推荐
```

一个语文教学游戏详情页实勘可见：
- 主体是**大尺寸 iframe 预览区**，可以直接试玩课件。
- 右侧/下方固定出现**作者昵称、作者 ID、个人简介、关注按钮**。
- 操作区同时给出 **分享 / 一键改编 / 下载** 三个动作。
- 下方的"相关推荐"不是纯猜你喜欢，而是把**同一作品的多个改编版本**单独列出来，再混排泛内容推荐。

这说明飞象把详情页设计成了一个明确的转化漏斗，而不是单纯的内容消费页。用户在这里至少有四种后续动作：关注作者、分享链接、下载成品、改编再生产。尤其是**"同题改编链"被单独展示**，等于公开告诉用户：这个作品不是终点，而是模板起点。

**更重要的判断**：飞象把"改编关系"产品化了。不是只有内部知道某个课件被改编了几次，而是直接把"改编《…》"作为相关推荐的一类显式内容暴露出来。这会进一步强化教师的认知：最自然的用法不是从零做，而是沿着优秀样例继续改。

### 3.4 资源广场的指标不是一个数，而是两套心智

继续查看页面后，一个之前没写透的点是：资源卡片和详情页里同时出现了**大号热度数**和**另一组较小的互动数**。例如广场卡片上会同时看到 `6.5万 / 6506`、`7.8万 / 6196` 这类双指标；详情页上也有 `5.8万 / 4290` 这样的组合。

虽然当前没有直接接口文档可证明字段定义，但从页面分布和 CSS 命名看，前一个大数更像**浏览/曝光心智**，后一个数更像**收藏/点赞/站内认可心智**。这两个数并列展示的产品意义很明确：

- 大数负责制造"大家都在看"的从众感，降低首次点击犹豫。
- 小数负责表达"这不是空流量，而是有人认可/保存"的质量感。

这比只展示一个浏览量更有效。教育内容天然存在"有人看，不代表值得用"的问题；把曝光和认可拆成两套数字，能让教师更快判断"这是热闹"还是"这是好用"。

## 4. 核心创作流程与交互范式

这是飞象产品力最集中的地方。一次实测案例：**上传 71.5KB 九年级物理复习 Word → 输出 20 页课件**，全程 8+ 阶段、2 个用户确认门控、9 个 Batch 串行生成。

**端到端阶段（产品视角）**：

```
① 信息确认表单（AI 预填默认值）
   ├ 课时内容（单选 + 自定义）
   ├ 年级刻 / 册次刻 / 教材版本刻（单选组，"刻"是飞象自造术语后缀）
   ├ 教学重点刻（多选，默认全选）
   ├ 课时数刻（1/2/3 课时以上）
   └ 特殊要求（自由文本）
   →【用户确认门控①】点"确认/不修改继续"，出现"已确认"标签
② AI 专家思考（自动执行，可折叠卡片）
   🔍关键词提取 → 📖研读课程标准 → 📚教材内容检索（无资料时橙色告警）
   → 📖精选学科资源（外链 PDF）→ 🔍目标与重难点刻 → 🔍逐页内容设计（表格）
   →【用户确认门控②】审阅大纲、补充需求
③ 课件生成计划（列出 4 个强互动页 + 16 普通页 + 9 个 Batch 划分）
④ 分批生成进度（9 条进度行，逐条 🔄 → ✓）
⑤ 交付物卡片（绿色 HTML 图标 + 文件名 + 缩略图预览 + ↓下载）
⑥ AI 说明文字（功能亮点 + 操作指南）
⑦ 追问修复闭环（用户绿色追问卡片 → 定向修复 → 新版交付物卡片 + 修复摘要）
```

### 4.1 每个设计决策背后的逻辑

**结构化参数 + AI 预填默认值**

决策逻辑：让教师从"输入者"变成"确认者"。认知负担从"我该写什么"变成"这个对不对"，难度差距极大。对于高频使用的教师，大多数情况下默认值就够用了，直接点确认即可。

代价：预填质量决定了这个设计的成败。如果 AI 经常预填错年级或教材版本，教师会养成"不看就改"的习惯，反而增加摩擦。这对背后的意图理解模型要求很高。

**专家思考过程可视化**

**这是飞象在教育场景里最聪明的信任建设设计。** 为什么在教育场景里"展示推理过程"尤其关键？因为教师对内容的权威性要求极高——他们不会接受一份"不知道从哪里来"的课件。飞象的专家思考把 AI 的工作流程透明化：「我先研读了课程标准→检索了教材→列出了重难点→设计了每页内容」。这个过程让教师能够评估"AI 是在认真做教研，还是在瞎编"。

默认折叠的设计同样精妙：熟练用户可以跳过，新用户需要建立信任时可以展开。这是一个"不打扰但随时可审计"的透明度设计，值得在 ClassIn 的生成流程里借鉴。

**2 个 HITL 门控的代价分析**

在"确认参数"和"确认大纲"两个节点设置门控，是一个明确的废稿率 vs 流畅度权衡。增加了两次中断，牺牲了"一键到底"的爽感；换来的是教师对产物的掌控感，以及"生成出来的课件和我想的是同一个方向"的概率。

什么时候不该用 HITL？当用户是高频用户、需求非常明确、生成速度是第一优先级时。飞象在"复杂度感知路由"里对此做了部分处理——简单任务走单次生成快通道，不需要等 HITL 确认。

**分批生成 + 实时进度的反焦虑价值**

20 页课件生成可能需要数分钟。9 条「🔄→✓」进度行是核心的反等待焦虑设计——它把一个不透明的等待，变成了一个可观察的、逐步推进的任务序列。用户不是在等一个黑盒，而是在看一个工作中的生产线。这种"进度可见"本身就产生了心理上的掌控感，显著降低了中途放弃的概率。

**追问修复闭环的信任意义**

"改了什么"透明化（AI 输出修复摘要 3 条）是建立长期信任的关键动作。一次生成后教师追问"修复公式渲染"，AI 不只是重新生成，还明确告诉教师"我改了这三件事"。这让教师能判断 AI 是否真的听懂了，也让下次追问有更清晰的预期。这是从"一次性工具"到"可信赖助手"的关键行为差异。

### 4.2 交互范式地图

| 范式 | 设计意图 | 好处 | 代价 / 边界 |
|---|---|---|---|
| 对话即应用（Chat as Application） | 用聊天范式装复杂多阶段流程 | 零学习成本 | 对话流过长时历史感消失 |
| 结构化参数 + AI 预填 | 认知负担从"输入"转到"判断" | 大幅降低开始成本 | 预填错了反而被跳过，对默认值质量要求高 |
| 专家思考可视可折叠 | 打开 AI 黑盒，建立教育场景信任 | 增信任、不打扰 | 展开内容过多会让教师觉得繁琐 |
| Plan→Confirm→Execute + HITL | 规划确认再执行 | 降废稿率、给掌控感 | 多两次交互，牺牲一点爽感 |
| 分批生成 + 实时进度 | 反长等待焦虑 | 心理掌控感 | 9 条进度只是感知优化，实际等待时间不变 |
| 强互动/普通页分级 | 质量预期透明，成本可控 | 用户不失望、资源不浪费 | 强互动稀缺投放，无法每页都强互动 |
| 追问修复闭环 | "改了什么"透明化 | 建长期信任 | 需要 AI 准确理解哪部分要改 |
| 示例提示词 + 热度 | 从众效应、降门槛 | 新用户不知道怎么输入时有参考 | 热门示例可能引导用户往同质化方向走 |
| 改编优先 | 把广场内容变成生产起点 | 降门槛、形成内容飞轮 | 过度依赖改编可能导致内容同质化 |

## 5. 信息架构与导航设计

飞象用一套**三层导航**承载全站，层次分明：

1. **左侧栏（功能工具 + 任务管理）**：固定约 240px，含"新建任务"CTA、4 个 AI 工具、置顶/今日/历史任务、底部积分与用户名。
2. **顶部学科分类（资源广场）**：横向 Tab，12 类（推荐/语文/数学(热)/英语/物理/化学/信息科技/课堂工具/学前教育/高等教育/职业教育/其他）。
3. **二级筛选器**：点学科后展开，教材版本 × 年级 × 学期三段筛选。

**"对话即文档"的持久化策略**：历史任务以"标题"为索引常驻左栏，点击可回到该对话继续编辑/追问。这把飞象从"一次性生成工具"变成了"教师的备课工作台"——劳动成果被沉淀，不是用完即弃。

这也是支撑"改编优先"的前提：要改编别人的作品，先得能找到自己历史生成的起点。

**代价**：以"对话"为最小单元做持久化，缺乏资源库级的协作、版本治理与跨人共享。一份课件被追问修复了三次，产生了三个版本，但这三个版本都只存在于对话历史里。如果要在团队里分享"我的最佳版本"，或者回滚到上一个版本，目前的架构没有干净的路径。

### 5.1 作者主页不是个人页，而是"内容货架"

继续进入作者主页后可以看到，飞象没有把作者页做成社交资料页，而是做成了**面向复用的作品货架**。典型结构是：

1. 顶部作者名、简介、关注按钮。
2. 中部三 Tab：**全部 / 课件 / 收藏**。
3. 下方作品网格，以封面缩略预览 + 标题 + 作者名 + 指标数组织。

值得注意的是，作品卡片缩略预览不是静态封面图，而是直接用 **iframe 缩略嵌入** 课件内容。这意味着：
- 作者页天然支持展示"这个作者擅长哪一类交互样式"，而不只是展示标题。
- 用户进入作者页后，本质上是在浏览一个**可复用模板集**，而不是看一个人的社交资料。

实勘中一个英语作者主页里，大量标题都呈现明显的"参数化批量生产"特征，例如"生成英语词汇句型教学网页课件"、"生成互动英语句型操练课件"、"生成指定短语句型互动教学课件"。这说明飞象社区里已经出现了**个人模板工坊型创作者**：他们不是偶尔产出一份课件，而是在某个学科垂类里持续生产可改编底稿。

## 6. 产品策略：能力边界与战略判断

### 6.1 飞象的天花板在哪里

飞象的能力边界不是"生成质量不够好"，而是它的商业模型和技术架构共同决定的：**止步于"生成 + 交付一个独立 HTML"**。

具体表现：
- 无"推送到课堂/发布作业/分享班级"动作入口
- 不回收学生作答数据（课件内 postMessage 上抛但宿主不消费）
- 无学情绑定（生成不感知班级历史表现）
- 无 SCORM / xAPI，无法被 LMS 追踪
- 与任何教学平台零集成

**这不是功能缺失，是战略选择**：开环换来了最低的集成成本（任何教师下载即用，不依赖任何平台）和最快的用户覆盖速度。飞象选择了做广度，放弃了深度。

### 6.2 飞象真正做不了什么

因为选择了开环，以下三件事飞象永远做不了：

**1. 据学情个性化生成**：飞象不知道这位教师的班级里哪些学生在哪个知识点上薄弱，因此生成的课件只能是"通用版"，而不是"基于 3 班上学期学情定制版"。这是数据闭环产品最核心的竞争力来源，也是飞象永远无法触达的。

**2. 内容质量迭代**：飞象不知道哪些互动设计学生真的在用、哪些题型错误率高、哪个教师的课件最有效。没有这些数据，就无法判断生成质量在提升还是在退步，也无法做有针对性的 Prompt 优化。飞象只能依赖主观体验打磨，而不是数据驱动迭代。

**3. 形成平台价值**：飞象的商业价值完全体现在"单次生成"上，教师离开飞象后，生成的课件和飞象没有关系了。这意味着飞象很难形成"网络效应"——教师A 用飞象教课产生的价值，对飞象平台没有反馈。而一个能收回作答数据的平台，随着使用时间越长、数据越多，对所有用户的价值越大。

### 6.3 软规范的一致性风险

课件内层设计靠 Prompt 约束而非硬资产，跨样本可见"令牌名一致、取值由 LLM 现场决定"。这带来的风险不只是"颜色可能不一样"：

**跨版本漂移**：今天的 Prompt 约束出来的课件风格，和六个月后升级 LLM 版本后的风格，可能差异显著——而用户没有任何感知，也没有版本快照可以对比。

**无法回滚**：如果某次 Prompt 更新导致生成的课件质量下降（比如交互一致性变差），没有"上一个好版本"的硬资产可以回退到。这对需要品牌一致性的教育产品来说是真实风险。

**无法质检**：因为没有 schema，无法做"这份课件的所有单选题都有正确答案标注"的自动化验证。每次生成后靠人工打开课件检查，规模上不去。

# 第二部分 · 技术侧

> **研发视角一句话（TL;DR）**：飞象 = **React SPA 壳（Hash 路由 + featureId）** + **多阶段 agent/workflow 编排（HTTP 短轮询拉状态，非 SSE/WS）** + **LLM 从零生成自包含 HTML 课件** + **工程化外壳（iframe srcdoc 宿主 + 埋点 + 共享库 CDN + 发布后处理噪声）**。
> 
> **关键判断**：只有"外壳"被工程化，"内层课件内容"是 LLM 每次现写、零复用、无 schema——这一架构起点决定了它能"演示任意互动"但"天然采不到结构化作答数据"。

## 1. 前端架构与运行时

### 1.1 主站（feixianglaoshi.com）

| 项 | 结论 |
|---|---|
| 架构 | **SPA**，前端渲染（`document.body.innerHTML` ~182KB） |
| 路由 | **Hash 路由** `#/home?featureId=20`，`featureId` 切功能 |
| 框架 | **React**（推断：Ant Design 类名 + CSS Modules 哈希；生产构建未挂 `window.React` 全局） |
| UI 库 | **Ant Design**（DOM 中可见 `.ant-menu-item` / `.ant-layout-sider`） |
| 样式 | **CSS Modules**：大量 `xxx_哈希` 类名（`content_gQ3o8`、`sidebarTools_Pra2U`、`historyItem_Nzu0A`） |
| 构建/分包 | Webpack chunk（`main.bc81b10d.js`、`663.d8f7e7c1.chunk.css`），静态资源带 `?v=2` |
| 公式渲染 | **MathJax**（`tex-chtml.js?v=2`） |
| 主站埋点 | **神策 Sensors Analytics**（`sensorsdata_app_call_js`、`__track_config`） |
| 分享详情页增强脚本 | `metis-lectio/*.js` + 一整套 MathJax 扩展脚本同时挂载 | 实证 |

**技术选型判断**：React + Ant Design + CSS Modules 是保守、成熟、招人友好的组合，对一个需要快速迭代、堆叠大量表单和列表的 B 端工具型产品是合理选择。Hash 路由回避了服务端路由配置，部署简单。这套选型没有任何惊喜，但工程决策不需要惊喜——它需要的是稳定。

### 1.2 课件产物运行容器

| 项 | 结论 |
|---|---|
| 容器 | `<iframe srcdoc="…整份 HTML（实体编码）…">` 沙箱嵌入 |
| 详情页预览 | `iframe.musk-html-preview-iframe`，`sandbox="allow-scripts allow-same-origin"` |
| 作者页缩略预览 | `iframe.musk-html-thumbnail-iframe`，直接嵌入课件缩略内容 |
| 数据注入 | 外壳注入 `__SERVER_DATA__`：`watermarkChatId`（对话式生成）、`watermarkBizId`（多租户）、`watermarkUrl`、`watermarkEnv` |
| 自标记 | `<meta name="generation-source" content="AI生成">` |
| 重定向 | `file://` 打开自动重定向到在线地址 |

**为什么用 iframe srcdoc 而不是独立 URL 加载**：`srcdoc` 把整份课件 HTML 作为字符串注入，实现"自包含 + 沙箱隔离"——课件内层的脚本污染不了主站，主站通过外壳注入水印与多租户标识。`watermarkChatId` 说明每次生成都绑定一个对话，这是审计和内容溯源的基础。

**这个选择的代价是什么**：iframe 边界天然阻断内层与宿主的数据互通。课件内的任何 `postMessage` 上抛，宿主不接收就是数据黑洞。这是后文"作答数据断流"的物理根因——不是缺一行代码，而是架构上的边界。

**这次继续实勘补到的一个细节**：详情页预览 iframe 的沙箱是 `allow-scripts allow-same-origin`。这说明飞象在"可运行复杂交互"和"保持宿主隔离"之间做了一个很工程化的中间态选择：允许课件内部完整执行脚本与资源访问，但仍把它关在 iframe 容器里，不让主站和课件 DOM 直接混在一起。作者页里的缩略预览则进一步把这套 iframe 机制下沉到社区列表层，形成"所见即所得"的作品货架。

## 2. Agent / Workflow 编排全流程

实测案例：**上传 71.5KB 九年级物理复习 Word → 输出 20 页课件**，全程 8+ 阶段、2 个 HITL 门控、9 个 Batch 串行生成。

```text
①意图理解&教学设计 → ②大纲/分页规划(outline) → [用户确认门控①]
  → ③专家思考(~2000字) → ④逐页大纲设计 → [用户确认门控②]
  → ⑤图片生成(文生图) → ⑥复杂度评估(强互动 vs 普通)
  → ⑦分批串行生成(9 Batch + 注入骨架) → ⑧素材回填 → ⑨互动/状态装配
  → ⑩后处理(噪声/水印) → ⑪外层壳封装(iframe + 埋点) → 交付物卡片
  → [追问] → ⑫增量定向修复 → 新版交付物卡片
```

### 2.1 关键技术决策一：为什么把"强互动页"和"普通页"分开规格

第⑥阶段的复杂度评估不是可有可无的步骤，而是整个编排成本控制的核心。一次 20 页课件中，实测有 4 页被标记为"强互动"（需要 Canvas/anime.js/Three.js 等），16 页是"普通内容展示页"。

**为什么要分开**：

首先是**LLM 上下文与稳定性问题**。强互动页的生成需要编写复杂的 JavaScript（动画状态机、事件处理、Canvas 绘制逻辑），如果把强互动页和普通页混在同一个 Batch 里生成，LLM 在同一次调用里需要同时处理"动态演示逻辑"和"知识卡片排版"，上下文干扰大、出错率高。把强互动页单独占一个 Batch，让 LLM 只专注一件事，成功率更高。

其次是**成本分配问题**。强互动页的生成成本（token 数、重试概率）比普通页高 3-5 倍。如果把所有 20 页都做强互动，生成成本会爆炸，时间也会翻倍。飞象的设计是"在值得强互动的地方投资强互动"——抽象动态过程（四冲程、电路连接、物理量关系）才上重互动；概念定义、公式卡片只用静态展示即可。这是资源配置的合理决策。

第三是**质量预期管理**。在生成计划阶段（第③步）就告诉教师"第6页是强互动、第12页是普通页"，教师对产物有明确预期，不会在看到第12页只有静态卡片时失望。

**对 ClassIn 的启示**：如果做题型生成，同样应该把"复杂交互题"（拖拽连线、动态推演）和"普通选择/填空题"分开处理，分配不同的生成 Batch 和 Prompt 策略，而不是一刀切。

### 2.2 关键技术决策二：交互剧本中间层解决了什么问题

飞象在强互动页生成前，有一个明确的"交互剧本"编写步骤——把"教学需求"翻成"技术方案"。以热机四冲程页为例：

```
教学需求：演示四冲程发动机的工作过程
        ↓ 交互剧本
技术方案：4 个冲程按钮（吸气/压缩/做功/排气）+ 汽缸活塞动画（anime.js 控制绝对定位 DIV）
         + 冲程名称和说明文字同步切换 + 完成全部冲程后显示能量转化小结
        ↓ 代码生成
实际 JS：runStroke(index) 函数，strokes[] 数组驱动，每步更新 DOM + 触发 anime.timeline()
```

**为什么不直接从教学需求跳到代码**：

LLM 在"直接从自然语言生成复杂互动代码"时，上下文边界不清晰，容易生成"看起来有互动"但功能不完整或交互逻辑混乱的代码。加入"技术方案"这一层，相当于让 LLM 先把需求结构化（明确交互组件、数据流、事件触发），再基于结构化方案写代码——两步的可控性远高于一步。

这也解决了**一致性问题**：技术方案层明确了"用 anime.js 做动画"、"用 Canvas 还是 DIV"，后续的代码生成有了明确约束，不会每次乱选库。

**这一层不能省略的场景**：有多个组件需要协同（按钮+动画+文字同步）、有状态机逻辑（正确/错误/完成）、有物理或逻辑验证（电路通断判断）的交互页，都必须经过交互剧本层。直接生成会导致组件间耦合混乱、debug 代价极高。

### 2.3 能力体系 = Skill（Prompt 约束规则，非代码组件库）

一份 Skill 含：角色定义、CDN 库声明 + 触发条件、CSS 变量强制、交互模式指南、壳框架兼容规则（白名单类名）、单/多页输出格式约束、**反噪音约束**（禁止 AI 幻觉 build marker/GA4/CMS）、学科正确性约束。

**一个值得注意的矛盾**：生成阶段 Skill 明确禁止 AI 生成幻觉元数据（build marker、GA4 标识、CMS 时间戳），但发布阶段的后处理流水线会主动注入大量假元数据（`WCAG 2.1 AA`、`Webpack 5.88.2`、假 GA4 measurement id、离屏 Marker 水印）。两个方向截然相反。

**解读**：生成阶段的反噪音是为了让 LLM 专注于真实内容，不要在课件里自行添加与课件逻辑无关的代码；发布阶段的加噪音是为了反爬虫/反逆向——把真实代码淹没在大量无意义的假元数据里（真实代码 <15%），让试图逆向拆解飞象课件的人增加成本。这是一个有趣的工程博弈。

## 3. 流式机制：为什么选短轮询而不是 SSE

> **agent runner 用 HTTP 高频短轮询拉执行状态，不是 SSE / WebSocket。**

| 项 | 结论 | 证据 |
|---|---|---|
| 机制 | **HTTP 短轮询** | 实证，CDP 网络监控 |
| 端点 | `/musk-ai-studio/api/agent-chat/executing-status` | 实证 |
| 频率 | 约 **80–150ms / 次** | 实证 |
| 文字呈现 | **整块出现**，非逐字流式 | 实证 |

### 3.1 SSE 在长任务场景的具体风险

SSE（Server-Sent Events）在实时流式场景（如 ChatGPT 逐字打字效果）是常见选择，但在飞象这种**分钟级、多阶段批量生成**场景里，SSE 有几个具体风险：

**网关超时**：大多数 API 网关（Nginx、AWS ALB、CloudFront）默认的连接超时在 60–300 秒之间。20 页课件的生成时间可能超过这个范围，导致网关主动断开 SSE 连接。客户端需要处理重连逻辑，并在重连后从正确的阶段恢复——状态恢复设计复杂，出错时用户体验极差。

**反向代理与 CDN 的缓冲问题**：许多反向代理会对 HTTP 响应做缓冲（Buffer），导致 SSE 的流式数据被积累后批量推送，实际上失去了"实时"效果。需要在所有中间层配置禁用缓冲（`X-Accel-Buffering: no`），维护成本高。

**多阶段状态机的语义不匹配**：SSE 的设计范式是"服务器持续推送增量数据"，但飞象的生成是"离散的阶段切换 + 每阶段内的异步任务"。把一个多阶段状态机套进 SSE 的流式范式，需要设计复杂的事件类型协议；出错或重连时的状态恢复更难处理。

### 3.2 短轮询的代价与规模瓶颈

短轮询的代价是真实的：

**QPS 压力**：80–150ms 一次，意味着每个活跃用户每秒产生约 7–12 个轮询请求。如果有 1000 个用户同时在生成课件，`executing-status` 端点需要承受 7,000–12,000 QPS。这个端点本身逻辑简单（查任务状态），可以做到极轻量，但依然需要良好的缓存和水平扩展设计。

**在什么规模下会成为瓶颈**：当同时活跃的生成任务超过 5,000–10,000 个，轮询 QPS 会对服务器产生明显压力，需要考虑：① 动态调整轮询频率（任务刚提交时频繁，进入等待队列后降频）；② 引入长轮询（Long Polling）作为过渡——客户端发请求后服务器 hold 住 30 秒再响应，减少无效轮询次数；③ 对有条件的客户端升级到 WebSocket。

**现在合不合理**：飞象当前规模下（教育工具类 SaaS，不是高频交互平台），短轮询是合理的。它的稳健性优势（无状态、易水平扩展、天然容错）对长耗时任务是实际价值，不是偷懒。

⚠️ **注意**：部分分析资料里提到"SSE streaming 响应"，与本节实证冲突。以 CDP 直接观测到的短轮询为准，SSE 是行业惯例推断，研发引用时须独立抓包确认。

## 4. 课件内容结构：无 schema 的完整后果

### 4.1 两种交付形态

- **单页课件**：一份完整 HTML 塞进 `iframe srcdoc`，自包含。
- **多页课件**：`page-shared`（设计令牌 + 共享库）+ 20 个 `<template class="page-data" data-id data-name>` 分页 + 壳框架翻页脚本（手势/键盘/缩略图/演示模式）。

### 4.2 内容数据结构 = 无 schema / 无 DSL（★这是整份报告最重要的技术判断）

内容就是 HTML 代码本身，**没有"题目对象/选项数组/答案字段"这一层**：

- 单选题：`checkQ7(this, true/false)` —— **正确答案以布尔字面量硬编码在 `onclick`**，题干/选项/解析全是 DOM 文本。
- 分步揭示：`showStep()` 计数器自增。
- 动态演示：`runStroke(index)` 数据数组驱动绝对定位 DIV + anime（**非真 `<canvas>`**）。
- 拖拽建模：`testCircuit()` 仅查"元件是否齐全"，**不验真实连线拓扑**。
- 拖拽归类：`handleDrop()` 靠 `data-type` 字符串相等匹配。
- **零组件抽象**：同名 `saveState()`/`state` 在多页各写一份，靠 `<template>` 隔离作用域。

### 4.3 无 schema 的后果不只是"采不到数据"

通常说"飞象采不到作答数据"停在结论层，但背后的具体影响更深：

**影响一：无法做内容版本管理。** 一份课件改了之后是"新版"，但新版和旧版的差异在哪里？没有结构化内容表示，就无法做 diff 比较，也无法回滚到某个具体的"好版本"。每次追问修复产生一个新 HTML 文件，版本之间的关系只能靠时间戳，不能靠内容语义。

**影响二：无法做质量回归测试。** 今天的 LLM 版本生成的单选题，所有正确答案都在 `onclick="checkQ(this, true)"` 里。如果下个月换了 LLM 版本，怎么自动化验证生成质量没有下降？没有 schema，就没有机器可解析的"这道题的答案是什么"——只能靠人工打开每份课件点一遍。在大规模生成场景里，这是质量保障的致命缺陷。

**影响三：无法做 A/B 测试不同交互效果。** 飞象现在无法知道"拖拽连线题 vs 点击选择题，哪种形式的学生完成率更高"。要做这个实验，需要能区分题目类型、能追踪用户完成行为。没有 schema，两件事都无法做到。

**影响四：无法做个性化生成。** 个性化生成需要"上次这个学生在什么题型上出错率高"的结构化输入。飞象的数据管道里，输入是"教师描述的教学目标"，输出是 HTML。没有任何关于"学情"的结构化数据能进入生成链路，更不可能做"为这个班的弱项定制内容"。

**影响五：系统自己不知道内容里写了什么。** 最讽刺的一点：`checkQ7(this, true)` 里的答案，只有打开代码看才知道是 true 还是 false。飞象的系统本身无法机器读取"第7题的正确答案是哪个"。这意味着内容校验、答案一致性检查、题目去重，都只能靠 LLM 理解 HTML 代码，而不能靠结构化查询。

### 4.4 壳框架事件拦截机制

壳框架拦截页面空白处点击用于翻页，用**约定 CSS 类名白名单**自动放行可交互元素（`.tool-btn / .drag-item / .drop-zone / .option-btn / .input-box / canvas / video …`），并用 `MutationObserver` 监听动态元素。

AI 生成课件只要**遵守类名约定**即可被壳框架识别。这是"约定优于配置"的轻量运行时，**不是 schema 驱动的渲染引擎**——这个区分至关重要：约定是软规范，可以被 LLM 遵守也可以被 LLM 偶尔遗漏；schema 驱动意味着渲染引擎是代码层的强制契约，LLM 只填数据。

### 4.5 状态协议 = postMessage（有名无实）

课件内 `window.parent.postMessage({type:'saveState'|'restoreState', state}, '*')` 上抛，但 **state 形状逐页自定**（`{mode}`/`{currentStroke}`/`{items}`/`{correctCount}`），无统一 schema，无法跨页聚合；且 **parent 根本不消费 saveState** → 作答数据在 iframe 边界断流。

飞象其实"差一步"就能采到数据——课件内已经在 `postMessage` 上抛状态了。但因为：① state 无统一 schema（每页自定义，无法标准化解析）；② 宿主 parent 不监听不消费，这个"半成品的回收通道"形同虚设。这不是工程疏漏，而是强烈暗示**回收作答数据从来不是飞象的产品目标**。

| 能力 | 飞象现状 | 证据 |

| 页面停留时长 | ✅ 能（埋点 `use_time`，多事件 + 100ms 防抖累加） | 实证 |

| 与 LMS / 班级 / 课堂集成 | ❌ 无（生成与使用两孤岛） | 实证 |

**数据缺失的具体盲点**：飞象现在不知道：
- 哪些互动设计（拖拽 vs 点击 vs 填空）学生真的会用、哪些被跳过

- 哪个教师创作的课件使用时长最长、学生参与度最高（无法做"最佳实践推荐"）

**一个极具说服力的间接证据**：飞象把埋点投入**全压在"停留时长"这一个指标**上（多事件 + 100ms 防抖累加，做得非常精细），却对"作答数据"一字未采。一个连停留秒数都要精细累加的团队，不可能"忘了"采作答——这只能说明**采作答数据不在飞象的产品目标里**。这是飞象与"需做闭环采集"产品之间的根本分水岭。

## 6. 跨样本结论："软规范 vs 硬外壳"（★核心技术判断）

> 用 Grep 抽样 4 个课件样本（courseware / s2 / s3 / s4 的 `.inner.html`）交叉验证。

### 6.1 被工程化的"硬资产"（跨样本一致）

- **共享前端库**：`metis-misc/zgLDUdmazTYc0B4K6Cor.js`、`zZZY40t7WJC7UdQCPACm.js` 等同一组文件跨样本复用 → 库层是固定白名单。
- **外层壳 + iframe 宿主 + 埋点**：结构跨样本一致。
- **后处理噪声层**：`WCAG 2.1 AA`、`Webpack 5.88.2`、`Marker-xxxx`（`left:-9999px` 离屏隐写水印）、`data-measurement-id`(ga4)、`feature-flags`、统一 CSP 等假元数据成体系注入（真实代码 <15%）。

### 6.2 靠约定的"软规范"（跨样本各异）

- **设计令牌值逐课件不同**：`--primary` 在四样本分别是 `#3B82F6`(courseware) / `#2563EB`(s4) / `#01553D`(s3) / `#D7F324`(s2)，`--accent` 亦各异。**令牌名一致、取值由 LLM 现场决定**。`page-shared` 的 `:root` 仅定义 7 变量（`--primary/--secondary/--accent/--bg/--text/--success/--error`），但状态反馈色大量写死 hex（正确 `#dcfce7/#22c55e`、错误 `#fee2e2/#ef4444`），未复用令牌。
- **内层 DOM/CSS/JS 零代码复用**：每份课件各写各的，互动函数手写，命名约定相同但实现不共享。

### 6.3 软规范的具体风险举例

**令牌被旁路**：课件内层大量状态色（正确/错误/警告）写死 hex，没有通过 `--success/--error` 令牌，说明 LLM 在生成时"记住了名字但忘记用名字"。这种旁路在软规范体系里无法阻止，只能在 Prompt 里反复强调。

**跨版本样式漂移的具体场景**：假设飞象在某次 Prompt 优化里把"主色倾向改成更活泼的橙黄系"，那之前生成的所有绿色主色课件和之后生成的橙黄课件会共存于同一个广场，但没有任何方式知道哪个版本更受教师欢迎，也没有办法批量更新旧课件的颜色——每份课件都是一次性生成的 HTML。

**无法做编译期校验**：如果想验证"所有课件的主色都在品牌色范围内"，需要逐份课件解析 HTML 找到 `:root` 里的 `--primary`。而且解析 HTML 拿到的值是 LLM 当时随机决定的 hex，没有语义，无法自动化质检。

# 结尾 · 综合判断与对「ClassIn AI 助教升级」的对照启示

## 综合判断

飞象老师代表"AI 生成互动课件"这条**开环赛道的当前高水位**：它能用一句话或一份资料，经成熟的多阶段编排，产出 20 页带真实 Canvas/SVG 互动的专业课件，具备资源广场内容飞轮、追问修复闭环、积分商业化等完整产品形态。它的每一个工程取舍（React 保守栈、短轮询、整页 HTML、外壳工程化）都服务于同一个目标——**把"生成"这件事做到足够好、足够广、足够快**。

它的能力边界同样清晰：**止步于"生成 + 交付一个独立 HTML"**。这不是疏漏，是战略选择。无 SCORM/xAPI、不回收作答、无内容 schema、不与任何教学平台集成——这些"缺失"换来了最低的集成成本和最广的教师覆盖。飞象把全部赌注押在"生成侧"，主动放弃了"使用侧"与"数据闭环"。

| 维度 | 飞象（参照系） | 我方必须做的 | 为什么 |
|---|---|---|---|
| **内容结构** ★ | 无 schema，整页 HTML，零复用 | **schema/DSL 驱动**：题型组件 + 内容 JSON + 运行时 player | 这是能否做数据闭环的底层前提，见下方详述 |
| **闭环 vs 开环** ★ | 只埋停留时长，无作答回收 | **必须闭环**：SCORM/xAPI 采集 → 学情沉淀 → 反哺生成 | ClassIn 护城河，项目核心价值 |
| **运行时** | 翻页壳框架（约定类名放行点击） | **schema 驱动渲染引擎 + 行为埋点**，与 SCORM 运行时衔接 | 要采"哪题/答案/对错/用时"，需 player 统一接管 |
| **生成产物** | 独立 HTML，浏览即止 | **投放进 ClassIn 预习/课堂/作业/表单**，对接 LMS 活动体系 | 飞象不进教学场景，我方原生进场景 |
| **知识流向** | 单向：知识→生成 | **双向**：知识库供给生成 + 作答回流更新学情/反哺 | 飞象无"记忆系统"，我方闭合数据回路 |
| **个性化能力** | 通用课件，不感知班级学情 | **基于学情生成**：弱项重点强化、班级定制版本 | 这是飞象永远做不到的，是我方核心差异 |

### 三、为什么"必须用 schema/DSL"——不是技术洁癖，是产品地基

这一点值得对产品和研发都讲清楚，因为它决定了后续哪些功能能做、哪些做不了。

**如果 ClassIn 走"整页 HTML"路线（飞象路线）：**

- ❌ 无法做 SCORM/xAPI 采集（没有结构化题目对象，无法触发标准事件）
- ❌ 无法做作答分析（不知道"第3题答对了"，只能知道"页面停留了90秒"）
- ❌ 无法做个性化生成（无法把"学生错误率高的题型"作为结构化输入）
- ❌ 无法做内容版本管理（每次改动产生新 HTML，无法语义化 diff）
- ❌ 无法做 A/B 测试（无法区分不同题型/交互方式的效果差异）
- ❌ 无法做题库复用（每道题都是 HTML 片段，无法跨课件去重/复用）
- ✅ 唯一能做的：生成漂亮的可展示课件，教师下载拿走

**如果 ClassIn 走"schema/DSL 驱动"路线：**

- ✅ 结构化题目对象（题干、选项、答案、解析、题型、知识点标签）
- ✅ 运行时 player 统一接管交互（统一采集事件：开始、作答、提交、重试）
- ✅ SCORM/xAPI 标准输出（player 对接 LMS 运行时，自动上报分数和完成状态）
- ✅ 作答分析（题目级错误率、用时分布、尝试次数）
- ✅ 学情反哺生成（把"这个班第3知识点错误率 62%"作为生成 prompt 的输入）
- ✅ 内容版本管理（题目 JSON 可以版本化、可以 diff）
- ✅ 个性化生成（基于学情数据，为不同班级生成不同侧重的内容）

**结论**：schema/DSL 是一道起点就分叉、后期无法回头的架构选择。Phase 0 不立这个地基，后面所有数据闭环功能都是空中楼阁。这不是技术洁癖，是产品能力的底层前提。

### 四、我们的差异化在哪——让产品和技术都能讲清楚

**对产品说**：飞象做的是"AI 内容生成器"，我们做的是"AI 教学场景赋能平台"。区别在于：飞象的价值在"生成一次"，我们的价值在"生成一次、在真实课堂里用、把数据收回来、下次生成得更好"。随着使用时间增长，飞象的价值不变（每次生成都是新的），我们的价值随数据积累持续增长。这是工具和平台的本质区别。

**对技术说**：飞象证明了生成侧的工程范式（多阶段 workflow、复杂度路由、交互剧本、分批生成），这套东西是可借鉴的。我们的真正技术挑战在飞象没有的那一半：① 内容 schema 设计（能同时支撑 AI 生成和 SCORM 采集）；② 运行时 player（统一接管所有题型交互，标准化上报行为数据）；③ 学情数据管道（作答数据 → 知识点掌握度模型 → 生成 prompt 输入）。这三件事飞象没有做、也不会做，是我方真正要自研的核心。

**一句话差异化**：飞象开环，我们闭环；飞象止步于"下载一个 HTML"，我们把内容投进真实教学场景，并把学生的作答数据收回来反哺下一次生成。这不是"生成得更好"，而是"把生成这件事接进了教育价值链的完整闭环"。
