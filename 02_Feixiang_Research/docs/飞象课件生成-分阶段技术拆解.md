## 阶段 1 · 意图理解 & 教学设计

### 【做什么】
把"教什么、教给谁、教成什么样"翻译成可执行的课件骨架：学科、学段、知识模块、课型（复习课 / 新授 / 练习）、知识点编排顺序。

### 【从产物里看到的具体实现 / 证据】
产物本身不含规划过程，但 **18 页 `.page-container` 的排布**把"教学设计"的成品结构暴露得很清楚（`data-name` 直接是教研标签）：

```text
data-id  data-name        教学功能（从内容反推）
  1      封面             课题/学段/教材版本（人教版·九年级上册）   ← 全屏封面
  2      复习框架         双模块总览（热学 / 电学）              ← 先给地图
  3      分子动理论       知识回顾 + 扩散实验演示
  4      改变内能方式     做功 vs 热传递（双卡片对比演示）
  5      比热容与热量计算 公式卡片
  6      热机四冲程       强互动：Canvas/anime 四冲程演示
  7      热学概念辨析     选择题（单选判断）
  8      热量计算例题     分步解析（逐步揭示）
  9      电学模块导入     转场页（全屏渐变，无 page-container） ← 模块切换锚点
 10      两种电荷         知识卡片 + 配图
 11      串并联电路连线   强互动：拖拽建电路
 12      电压与电阻       知识卡片
 13      欧姆定律         公式（MathJax）
 14      电路分类练习     强互动：拖拽归类
 15      电学计算练习     例题 + 核对答案
 16      综合练习         热电综合
 17      易错辨析         "红灯区"易错点
 18      知识体系总结     全课知识地图
 19      核心汇总         公式/物理量速查表
 20      课后巩固         结课页（全屏居中）
```

可读出的**教学设计规范**（实证级别的结构特征）：
- **"总—分—练—结"四段式**：框架页（2）→ 分知识点讲解（3–6, 10–13）→ 练习（7,8,14,15,16,17）→ 总结（18,19,20）。
- **双模块对称**：热学（3–8）与电学（10–17）镜像编排，中间用**转场页（9）**做模块切换锚点。
- **强互动按知识类型投放**：抽象动态过程（四冲程 6、电路 11、分类 14）才上重互动；概念/公式页（5、12、13）只用静态卡片。

### 【生成规范推断】
- 存在一份"**课型 → 页型序列**"的模板规范：复习课 = `封面 + 框架 + N×(讲解|演示|练习) + 转场 + 总结`。
- 每页带一个**教研语义标签**（`data-name`），说明规划阶段产出的是**带语义的页清单**，而非纯页码。
- 强互动是**稀缺投放**（20 页里仅 3 页重互动），由"知识点是否含动态过程"触发——对应基线说的"复杂度感知路由"。

### 【对我们项目的启示】
- "课型→页型序列"这套**意图→结构映射**值得直接借鉴，作为我们生成编排的第一层 schema。
- 但飞象的"教学设计"成品**只活在 HTML 结构里、无法被机器复用**；我们应把它沉淀为**结构化 outline 对象**（页型枚举 + 知识点 ID + 互动类型 ID），为后续装配与数据采集留锚点。

## 阶段 2 · 大纲与分页规划（outline）

### 【做什么】
把教学设计落成可生成的**页骨架**：每页一个容器、统一标题样式、统一内容栅格。

### 【从产物里看到的具体实现 / 证据】
分页是**模板化**的，规律极强（`inner.html`）：

```html
<!-- 4632 行：唯一的全屏页（封面），不用 page-container -->
<template class="page-data" data-id="1" data-name="封面"> <div class="cover">…</div> </template>

<!-- 4658 行起：标准内容页，统一骨架 -->
<template class="page-data" data-id="2" data-name="复习框架">
  <div class="page-container">
    <div class="page-title">整体复习框架</div>
    …栅格内容…
  </div>
</template>
```

骨架三件套（实证，行号见上节）：
- **分片容器** `<template class="page-data" data-id data-name>`：每页一个，`data-id` 连续、`data-name` 是教研标签。
- **页容器** `.page-container`：18 个标准页统一使用；`.page-title` 统一为"32px / 700 / 左侧 6px 主色竖线"。
- **特例页** 不用 `.page-container`：封面（1）`.cover`、转场（9）`.transition-bg`、结课（20）行内 `justify-content:center`。

栅格几乎全靠**行内 `style` 写 flex/grid**（如 4661 行 `grid-template-columns:1fr 1fr;gap:30px`），**没有抽象出栅格类**。

### 【生成规范推断】
- 规划阶段产出的是 **"页清单 + 每页一个 `page-data` 模板"**，由统一骨架填充——这正是"分批生成 + 注入骨架"在产物层的落点。
- 标题、容器走**全局共享类**（见阶段 3），但**版式（栅格）逐页即兴**，说明 outline 只约束"页级骨架"，不约束"块级布局"。

### 【对我们项目的启示】
- "**统一页骨架 + 逐页分片**"的工程范式可借鉴（利于分批生成、断点续传、定向修复）。
- 但飞象的"分片"是 **`<template>` + 行内样式**，块级版式不可控、不可统计。我们应把分片升级为**结构化页节点（含 region/block schema）**，块级布局也走令牌化栅格，才能做到可质检、可复用。

## 阶段 3 · 设计系统与视觉规范（重点）

### 【做什么】
给每页注入统一的"设计护栏"：色彩令牌、字体、间距、组件基线样式、响应式策略、水印。

### 【从产物里看到的具体实现 / 证据】
**全部设计令牌集中在唯一的 `<template class="page-shared">`（4561–4630 行）**，是整份课件唯一的"全局 CSS"：

```css
/* 4578 行 :root —— 7 个设计令牌，全文件唯一一处定义 */
:root {
  --primary: #3B82F6;   /* 蓝（主色：标题、主按钮、强调竖线） */
  --secondary: #60A5FA; /* 浅蓝（主按钮 hover） */
  --accent: #F59E0B;    /* 橙（次强调：并联区、电压卡片） */
  --bg: #F8FAFC;        /* 页面底色 */
  --text: #1E293B;      /* 正文色 */
  --success: #10B981;   /* 绿（正确反馈） */
  --error: #EF4444;     /* 红（错误反馈） */
}
body { font-family: 'Noto Sans SC', sans-serif; color: var(--text); background-color: var(--bg); }
```

组件基线样式（同段，实证）：

```text
.page-container  min-height:100%; padding:30px 40px; display:flex; flex-direction:column;
.page-title      font-size:32px; font-weight:700; color:var(--primary);
                 margin-bottom:20px; padding-left:15px; border-left:6px solid var(--primary);
.card            background:white; border-radius:12px;
                 box-shadow:0 4px 6px -1px rgba(0,0,0,.1); padding:20px; margin-bottom:15px;
.btn             padding:10px 20px; border-radius:8px; font-weight:600; transition:all .2s; border:none;
.btn-primary     background:var(--primary); color:white;  /* hover → var(--secondary) */
```

**令牌的"使用纪律"很强但不完美**（实证）：
- 标题色、主按钮、强调线一律 `var(--primary)`（如 4601/4623/4604 行）。
- **但反馈色大量写死十六进制**而非用令牌：如 `checkQ7` 里正确态 `#dcfce7/#22c55e`、错误态 `#fee2e2/#ef4444`（4954–4964 行），并未复用 `--success/--error`。说明"令牌"是**生成提示词约束**，不是编译期强制，AI 经常旁路。

**字体与外链**：`Noto Sans SC`（4588 行），字体 `<link>` 在抓取时被替换为 `[unsafe-url-removed]`（4576 行，推断为 Google Fonts）。

**响应式策略（关键实证）**：内层课件**全程用固定 px**（标题 32px、卡片 padding 20px、汽缸 200×300px…），**内部没有 rem/vw/clamp/媒体查询**（搜 `vw|rem|clamp|@media` 仅命中噪声里的 `@media print` 和 `@media(max-width:0px)`）。真正的自适应在**外层壳**完成：

```css
/* 外层壳 22–28 行：iframe 容器吃满视口 */
.iframe-container { width:100vw; height:100vh; padding:56px 20px 20px; }
```
```js
/* 外层壳 7946 行：移动端才启用 rem 基准（375 设计稿） */
function adjustRem() {
  const pcBaseWidth = 375;
  const w = document.documentElement.clientWidth;
  document.documentElement.style.fontSize = `${(w/pcBaseWidth*100).toFixed(2)}px`;
}
```
→ **设计护栏 = "固定 px 画布 + 外层等比缩放"**，而非课件内部响应式。

### 【生成规范推断】
- 设计护栏以"**一段 `page-shared` 全局 CSS**"的形式**注入每份课件**：固定 7 个令牌名 + 4 个基础组件类（`page-container/page-title/card/btn`）。这就是 §4 基线说的"CSS 变量强制"在产物里的样子。
- 令牌**只约束骨架与主按钮**，状态反馈色靠 AI 即兴 → 规范覆盖不全，是**提示词级软约束**。
- 画布固定 px，自适应外包给壳——意味着课件被当作**固定比例的"页面卡片"**投放，不追求流式响应。

### 【对我们项目的启示】
- "**集中式设计令牌 + 共享组件基线类**"的护栏思路应直接采纳，但要**编译期强制**（lint / 主题变量注入），杜绝飞象式"令牌定义了却写死 hex"的漏网。
- 反馈色（对/错）必须令牌化——因为我们要**采集作答正误**，正误的视觉与数据要统一来源。
- 飞象"固定 px + 外层缩放"对纯展示够用；但我们的题型组件要进 ClassIn 多场景（预习/课堂/作业），建议**组件级响应式 + 容器查询**，不能只靠外壳缩放。

## 阶段 4 · 页面 / 组件生成（重点）

### 【做什么】
逐页生成 DOM + 局部样式 + 局部互动脚本，把每种"题型/互动"用手写函数实现。

### 【从产物里看到的具体实现 / 证据】
**每页自带一个 `<script>`，互动函数都是页内全局函数 + `onclick` 内联绑定**。逐控件举证：

**(1) 单选/判断题** —— `checkQ7(btn, isCorrect)`（4941–4966 行）
```html
<button class="btn" onclick="checkQ7(this, false)">A. 物体的温度升高，内能一定增加</button>
<button class="btn" onclick="checkQ7(this, true)">C. 任何物体在任何情况下都具有内能</button>
```
```js
function checkQ7(btn, isCorrect){
  const feedback = document.getElementById('feedback-7');
  feedback.style.display='block';
  if(isCorrect){ btn.style.backgroundColor='#dcfce7'; … feedback.innerHTML='✅ 正确！…'; }
  else        { btn.style.backgroundColor='#fee2e2'; … feedback.innerHTML='❌ 错误。请再仔细思考…'; }
}
```
- 命名：函数名直接绑题号 `checkQ7`；正确答案以**布尔字面量**硬编码在 `onclick` 里（`isCorrect=true/false`）。
- **没有"题目对象/选项数组/答案字段"**——题干、选项、答案、解析全是 DOM 文本与内联参数，**无法被结构化读取**。

**(2) 分步揭示（解析）** —— `showStep(s)`（4994–5000 行）
```js

  currentStep++; if(currentStep>3) return;
  const el = document.getElementById('step-'+currentStep);
  el.style.display='block';
  anime({ targets: el, opacity:[0,1], translateY:[10,0], duration:500 });

```
- 命名：步骤元素 `step-1/2/3`，靠**计数器自增**逐步显示；参数 `s` 实际未用（即兴痕迹）。

**(3) Canvas/anime 动态演示** —— `runStroke(index)`（四冲程，4876–4921 行）
```js

- 这是**全课最"工程化"的一处**：用**数据数组驱动 4 个状态**，DIV 模拟汽缸/气门/活塞（非真 `<canvas>`，是绝对定位 DIV + anime 改 `bottom/rotate`）。
- 同类还有第 4 页 `playWork()/playHeat()`（4757–4798 行，`anime.timeline()` 控活塞下压 + 着火）、第 3 页 `showDiffusion()`（4712 行，淡入配图）。

**控件命名规律（实证汇总）**：
```text
函数：动词+对象/题号   checkQ7 / runStroke / showStep / showDiffusion / testCircuit / playWork
元素 id：语义-序号     feedback-7 / step-1 / comp-battery / zone-series / piston-obj / valve-in
答案承载：             onclick 布尔字面量 | data-type 属性 | placedItems.includes 文本匹配
反馈：                 改 innerHTML/innerText + 内联背景色 + anime 缩放/淡入
```

**为什么没有组件抽象（实证）**：
- 同为"选择反馈""拖拽""anime 动画"，第 7、11、14、15 页各写各的 `saveState()`、各自的反馈 DOM、各自的颜色——**零复用**。
- 全文件出现**多个同名 `saveState()`/`state`**（4799、4923、5149、5320 行），靠 `<template>` 隔离作用域，**不是模块化，是"每页一份拷贝"**。

### 【生成规范推断】
- 题型/互动是**"提示词里的实现模式清单"**（按知识类型选择单选 / 分步 / 拖拽 / anime 演示），AI 现场把模式翻译成手写函数，而非调用组件库。
- 有一套**朴素命名约定**（`check*/show*/run* + id 语义-序号`），保证页内可读、可被壳框架识别（见阶段 6 `data-interactive`）。
- 互动追求"看起来对"，正确性按需简化（电路只查元件齐全）——与基线"演示型、非采集型"判断一致。

### 【对我们项目的启示】
- 飞象证明了"**手写整页互动**"表达力极强、上限高（四冲程演示相当精致），但代价是**零复用、不可统计、正确性靠注释妥协**。
- 我们必须**反过来**：把这 5 类控件做成**题型组件（schema 驱动）**——单选 `{stem, options[], answerKey, explanation}`、拖拽分类 `{items[{text,group}], zones[]}`、分步演示 `{steps[]}`、建模演示用参数化组件——这样答案是**字段而非 `onclick=true`**，正误是**组件内置而非内联 hex**，作答**天然可采集**。
- 命名约定可借鉴，但要升格为**组件契约（props + 事件）**，而不是页内全局函数。

## 对我们「AI 助教升级」的编排借鉴 vs 必须分叉

### 可直接借鉴的编排阶段（飞象已验证）

```text
意图理解  →  outline规划  →  设计护栏注入  →  分批/逐页生成  →  素材回填  →  装配  →  后处理/封装
  ✓借鉴       ✓借鉴           ✓借鉴(要强制)     ✓借鉴            ✓借鉴       △改造    △改造
```
- **意图→页型序列**：课型→页型骨架映射（总-分-练-结、双模块对称、转场锚点）可直接抄。
- **outline→统一骨架 + 逐页分片**：利于分批生成、断点续传、定向修复。
- **设计护栏 = 集中令牌 + 共享组件基线类**：思路照搬，但要**编译期强制**（飞象是软约束、常被旁路）。
- **复杂度路由 + 素材"代码优先、文生图兜底"**：节流规范可借鉴。
- **iframe srcdoc + 后端注入 + postMessage** 封装范式：适合投进 ClassIn 容器。

### 必须根本不同的地方（我们的护城河）

```text
                    飞象（开环）                      我们（必须闭环）
产物形态     整页HTML，每页手写互动、零复用      schema/DSL + 题型组件库 + 运行时player
答案承载     onclick=true / data-type / 文本    题型字段：answerKey/options/rubric
状态协议     postMessage state(逐页自定形状)     统一作答schema(题ID/作答/正误/用时/尝试)
正确性校验   学科准、交互"简化版/（略）"放行      契约测试：可采集+可判定+可恢复 才能装配
埋点         Frog use_time(停留秒数)            结构化作答事件 → SCORM/xAPI 回收
数据流向     单向；parent 不消费 saveState      双向；parent 消费作答 → 学情沉淀 → 反哺生成
设计令牌     软约束，反馈色写死 hex             编译期强制 + 反馈色令牌化(对接正误数据)
后处理       注入垃圾噪声抗爬                   干净结构 + 结构化水印(保采集可解析)
```

**三条最关键判断（给 PM + 研发）**
1. **分水岭在"产物结构"**：飞象的"整页 HTML + 内联 `onclick=true`"在纯展示课件上表达力极强，但**答案是代码字面量、作答在 iframe 边界断流**，结构上就**采集不到数据**。我们要数据闭环，就**不能走无 schema 路线**，必须 Phase 0 定下"内容 schema + 题型组件 + 采集运行时"。
2. **编排可抄、闭环自造**：意图→outline→护栏→分批生成→素材回填这条**生成流水线**飞象已验证，可大胆借鉴少走弯路；但 **作答采集 → SCORM/xAPI → 学情 → 反哺** 这条回路飞象**根本没有**（`saveState` 上抛却无人消费、Frog 只收 `use_time`），是我们必须从零设计、也是真正拉开差距处。
3. **护栏要硬、状态要统一**：飞象暴露的两个工程教训——设计令牌是软约束（反馈色写死 hex）、状态协议形状逐页自定（无法聚合）——正是我们要反向做硬的地方：**令牌编译期强制 + 作答状态统一 schema 且每个题型组件强制实现**。
```
