# 飞象外部资源获取与智能体链路深拆

> **目的**：把这次基于已登录 Chrome 的现场实勘、Playwright 页面读取、前端 chunk 逆向，收敛成一份专门回答“飞象如何拿外部内容、如何生成外部内容、前端如何接这些能力、背后是什么样的 agent/tool/runtime 分层”的技术说明。
>
> **证据口径**：
> - **【实证】**：来自真实页面 DOM、运行中网络资产、下载到本地的 HTML/JS chunk。
> - **【推断】**：基于接口命名、调用关系、运行时边界反推的架构判断。

---

## 0. 结论先行

飞象不是“一个大模型直接吐完整网页”这么简单。更准确地说，它是一个五层系统：

```text
输入层
  教师上传文档 / 图片，或在移动端通过微信能力拿图片与语音
    ↓
资源接入层
  file/chat/upload 等接口把原始文件标准化成 resourceId + resourceUrl
    ↓
对话编排层
  agent-chat/create / input / executing-status / submit-form-answer 负责任务状态机
    ↓
专项工具层
  ai-generation/text / ai-generation/image / chat-edit/optimize-page
    ↓
运行时层
  外层 preview wrapper + 内层 iframe 编辑器，通过 postMessage 把 AI 能力挂给页面
```

核心判断：

1. **外部资源获取**不是直接进 LLM，而是先进入统一资源层，变成 `resourceId + url`。
2. **图片能力**已经被做成独立工具接口，既可聊天阶段调用，也可在 HTML 编辑器内部二次调用。
3. **HTML 课件编辑器不是纯播放器**，它本身就是一个“工具宿主”，会把 AI 润色、AI 生图、图片上传、保存这些能力暴露给 iframe 内的内容页。
4. **音频链路目前更像移动端/微信桥接能力**，不是桌面 Web 主链路，也没有看到单独的“AI 音频生成”接口。
5. **飞象背后的智能体不是单体 agent，而是“编排 agent + 资源工具 + 专项工具 + runtime bridge”的组合。**

### 1.1 浏览器现场页

本次直接在用户已登录的 Chrome 中操作了 3 个真实页面：

- 对话编辑页：
  `https://www.feixianglaoshi.com/#/chat?chatId=2E919A2E67ACB7A7518B3BF0F67895DF&featureId=20&isHtmlEditing=1`
- 生成结果预览页：

- 首页/资源广场：
  `https://www.feixianglaoshi.com/#/home?featureId=20`

### 1.2 Playwright 直接读到的页面事实【实证】

对话页里存在两个隐藏文件输入：

- 附件输入：
  `.doc,.docx,.pdf,.ppt,.pptx,.xls,.xlsx,.txt,.csv,.html`
- 图片输入：
  `.jpeg,.jpg,.png,.webp`

两者都支持 `multiple=true`。

这说明飞象把“文档/附件”和“图片”当成两条不同的资源入口，而不是一个万能上传框。

### 1.3 预览页直接读到的事实【实证】

预览页外层 HTML 有这些关键特征：

- `<meta name="generation-source" content="AI生成">`
- `window.__SERVER_DATA__` 注入：
  - `watermarkChatId`
  - `watermarkEnv`
  - `watermarkBizId`
  - `watermarkUrl`
- 真正内容不直接写在 body，而是包在：
  - `<iframe class="content-iframe" srcdoc="...">`

内层 `srcdoc` 里又直接出现了这些注释和约束：

- `<!-- html-authoring:math-design palette=B-10 layout=L3 -->`
- `forbid=base64_images`
- `require=MathJax3,OrbitControls,Three.js,GSAP`
- CSP 允许的图片/媒体域名：
  - `*.fbcontent.cn`
  - `*.aliyuncs.com`
  - `storage.googleapis.com`

这意味着：

1. 生成内容并不倾向把素材内联为 base64。
2. 内容页被要求优先引用外部资源 URL。
3. 页面运行时对资源来源有白名单意识。

## 2. 外部资源获取总链路

### 2.1 文档/附件

#### 前端入口【实证】

对话页附件入口接受：

- Word
- PDF
- PPT
- Excel
- TXT
- CSV
- HTML

对应 accept：

```text
.doc,.docx,.pdf,.ppt,.pptx,.xls,.xlsx,.txt,.csv,.html
```

#### 上传接口【实证】

前端 chunk 明确调用：

```text
POST /musk-ai-studio/api/file/chat/upload
Content-Type: multipart/form-data
params: { featureId }
body: FormData(file)
```

#### 前端校验逻辑【实证】

从 `720.8e672c36.chunk.js` 可见：

- 图片文件最多 10 个
- 附件文件最多 10 个
- 单文件大小上限 20MB

错误提示也是前端内建的：

- 图片数量超过限制，最多 10 个
- 附件数量超过限制，最多 10 个
- 文件过大，超过 20MB

#### 上传后返回什么【实证】

上传成功后，前端会消费返回结果中的：

- `result.id`
- `result.url`

也就是把原始文件标准化为：

```text
resourceId
resourceUrl
```

这是飞象资源接入层最关键的抽象。

#### 资源后续怎么被用【推断 + 实证】

文档上传之后，agent 并不是直接吃浏览器里的 File，而是吃这个统一资源引用。后续生成流程里看到的都是：

- chatId
- resourceId
- resourceUrl

这说明后端先做了“资源入库/标准化”，然后编排层再引用资源，而不是让所有工具都自己处理原始文件流。

---

### 2.2 图片

图片在飞象里有三条不同路径。

#### 路径 A：用户直接上传图片【实证】

对话页图片输入接受：

```text
.jpeg,.jpg,.png,.webp
```

同样走：

```text
POST /musk-ai-studio/api/file/chat/upload
```

这说明“聊天输入图片”和“聊天输入文档”共用一个底层上传服务，只是前端入口和校验规则不同。

#### 路径 B：HTML 编辑器内部上传图片【实证】

HTML 预览/编辑 runtime 支持一个消息：

```text
IMAGE_UPLOAD_REQUEST
```

调用链路是：

1. 内层 iframe 发 `IMAGE_UPLOAD_REQUEST`
2. 外层 wrapper 收到后执行上传
3. 外层向后端发：
   `POST /musk-ai-studio/api/file/chat/upload`
4. 上传成功后回传：
   `IMAGE_UPLOAD_RESULT { url }`

这一段尤其重要，因为它说明：

- 内层内容页不能直接碰业务上传接口
- 真实上传能力在外层宿主
- iframe 内容页通过消息协议“借用”宿主能力

#### 路径 C：AI 生图【实证】

HTML 编辑器内部还支持：

```text
AI_GENERATE_IMAGE
```

收到后，外层调用：

```text
POST /musk-ai-studio/api/ai-generation/image
body: { prompt, referenceImageUrl? }
```

成功返回：

```text
data.result.imageUrl
```

再通过消息发回：

```text
AI_IMAGE_RESULT { url }
```

这表明飞象的图片链路不是单一路径，而是三分的：

1. 用户上传现成图片
2. 编辑器内部再次上传图片
3. AI 生成新图片

#### 图片最终存放在哪【实证】

真实页面和资源广场里看到了几类图片域名：

- 工作流图片：
  `https://musk-online.oss-cn-beijing.aliyuncs.com/pub-musk-ai-studio/workflow/file/picture/...`
- 用户/后台上传图：
  `https://musk-online.fbcontent.cn/pub-musk-ai-studio/user/upload/...`
- 头像：
  `https://musk-online.fbcontent.cn/pub-musk-account/headImage/...`

这说明飞象至少区分了：

- 工作流生成素材
- 用户上传素材
- 账户资料素材

从产品角度看，这种资源分仓是合理的，因为生命周期、鉴权、CDN 缓存策略都不同。

### 2.3 音频

#### 桌面 Web 主链路里没看到独立音频生成接口【实证】

这次真实页面、chunk、运行时接口里，没有抓到类似下面的端点：

- `ai-generation/audio`
- `tts`
- `speech`
- `voice-generation`

所以目前**不能把“飞象已经有 AI 音频生成主链路”当成事实**。

#### 但移动端/微信桥接能力是存在的【实证】

`196.7031d574.chunk.js` 是一整套微信 JS SDK 封装，明确暴露了：

- `startRecord`
- `stopRecord`
- `uploadVoice`
- `downloadVoice`
- `translateVoice`
- `playVoice`
- `pauseVoice`
- `stopVoice`
- `chooseImage`
- `uploadImage`
- `downloadImage`
- `previewImage`
- `getLocalImgData`

这说明飞象在移动端/微信容器里，至少准备好了：

1. 图片从相册/相机进入
2. 语音录制与上传
3. 语音转写

#### 对音频链路的判断【推断】

更像是：

- **桌面 Web 主流程**：文档/图片/HTML 编辑
- **微信端补充流程**：图片/语音采集与分享

而不是：

- 所有内容能力都围绕音频展开

换句话说，音频在飞象当前架构里更像“移动端输入工具”，不是“核心生成媒介”。

## 3. HTML 课件编辑器怎么接外部能力

这是这次最关键的新发现。

### 3.1 不是一个网页，而是双层 runtime

飞象生成的 HTML 课件不是直接把 AI 结果平铺出来，而是：

```text
外层 wrapper 页面
  ├─ 水印
  ├─ frog SDK 埋点
  ├─ 保存/编辑宿主逻辑
  └─ iframe srcdoc
       └─ 内层真实内容页/编辑页
```

### 3.2 双层之间靠消息协议通信【实证】

在 `720.8e672c36.chunk.js` 里直接能看到消息名：

- `INNER_IFRAME_LOADED`
- `EDITOR_READY`
- `HISTORY_CHANGED`
- `EDIT_END`
- `HAS_CHANGES_UPDATED`
- `SAVE_COMPLETE`
- `SAVE_FAILED`
- `AI_EDIT_REQUEST`
- `AI_POLISH_TEXT`
- `AI_GENERATE_IMAGE`
- `IMAGE_UPLOAD_REQUEST`
- `SHOW_TOAST`
- `OPTIMIZE_PAGE_REQUEST`
- `ADD_PAGE_REQUEST`

返回类消息：

- `IMAGE_UPLOAD_RESULT`
- `AI_POLISH_TEXT_RESULT`
- `AI_IMAGE_RESULT`

### 3.3 这意味着什么

这意味着内层页面并不是一个“完全自由的 HTML 文件”，而是默认知道自己跑在飞象宿主里，并且会把一些能力请求上抛给宿主：

- 我要润色这段文字
- 我要生成一张图
- 我要上传一张图
- 我要保存当前 HTML
- 我要优化这一页

宿主收到这些请求，才真正调用后端接口。

### 3.4 保存链路【实证】

HTML 保存调用：

```text
POST /musk-ai-studio/api/agent-chat/update-html-resource
body: { chatId, resourceId, htmlContent }
```

这说明飞象对 HTML 产物的抽象是：

- 一个 chat 任务
- 绑定一个 HTML 资源
- 可被覆盖更新的 `htmlContent`

所以它不是把网页当最终静态文件一次性扔出来，而是把 HTML 当“可回写的资源对象”。

### 3.5 AI 润色链路【实证】

编辑器收到 `AI_POLISH_TEXT` 后，会构造 prompt：

- 公式模式：
  `原LaTeX公式：... 修改要求：...`
- 普通文本模式：
  `原文：... 修改要求：...`

然后调用：

```text
POST /musk-ai-studio/api/ai-generation/text
body: { prompt, originalText }
```

再回传：

```text
AI_POLISH_TEXT_RESULT
```

这说明文本润色并不通过“大聊天主 agent”转一圈，而是直接走专项工具。

### 3.6 AI 生图链路【实证】

编辑器收到 `AI_GENERATE_IMAGE` 后调用：

```text
POST /musk-ai-studio/api/ai-generation/image
body: { prompt, referenceImageUrl? }
```

成功后回：

```text
AI_IMAGE_RESULT { url }
```

这和文本润色一起，构成了编辑器内部的“两大专项工具”。

### 3.7 页面优化链路【实证】

chunk 里还能看到：

```text
POST /musk-ai-studio/api/chat-edit/optimize-page
POST /musk-ai-studio/api/chat-edit/optimize-page/poll
```

这说明飞象已经有“页面级优化”能力，而且采用：

- 发起优化
- 轮询结果

这条链路很像一个局部重写工具，而不是全量重新生成。

## 6. 背后的智能体设计应该是什么样

下面这部分属于架构推断，但证据已经比较扎实。

### 6.1 编排 agent 不是直接做所有事

从接口分层看，飞象的总 agent 更像 orchestration agent，只负责：

- 接收用户需求
- 维护 chat 状态
- 决定当前该调用什么能力
- 把结果组织回对话流

而不是自己直接：

- 处理文件上传
- 生成图片
- 局部润色 HTML
- 持久化 HTML

### 6.2 它背后至少有 4 类工具

#### 工具 1：资源接入工具

- `file/chat/upload`

职责：

- 接原始文件
- 做类型/大小/数量校验
- 生成统一资源引用

#### 工具 2：内容生成工具

- `ai-generation/text`
- `ai-generation/image`

职责：

- 文本润色、局部重写
- 图片生成、参考图变体

#### 工具 3：HTML 编辑工具

- `update-html-resource`
- `optimize-page`

职责：

- 持久化 HTML
- 对 HTML 做局部优化

#### 工具 4：分发工具

- `share`
- `publish`
- `add-to-mine`
- `generate-tag`

职责：

- 把生成结果包装成可传播内容

### 6.3 runtime 本身也是 agent 的工具宿主

这是飞象最像“产品级 agent 系统”而不是“单纯网页生成器”的点：

- 内层页面提出能力请求
- 外层 runtime 接请求
- runtime 再替页面调工具

也就是说，**runtime 不是被动播放器，而是一个前端侧 tool router**。

这个设计的价值是：

1. 内容页可以更自由
2. 工具权限留在宿主，不暴露给任意 HTML
3. 页面编辑能力可以持续扩展，不必每次重造整页协议

### 6.4 音频/微信桥接是边缘工具，不是核心 agent 主干

目前更像：

- 当用户在微信环境中，用微信能力补充图片/语音输入
- 这些媒体再回到统一资源层

而不是：

- 音频本身是飞象课件的主输出对象

## 7. 对飞象产品能力边界的重新判断

### 7.1 它强的不是“网页代码生成”本身

真正强的是这三件事：

1. **资源统一化**
2. **专项工具化**
3. **runtime 宿主化**

这三者组合，才让飞象能做：

- 从聊天上传文档
- 在 HTML 里继续调图、改字、保存
- 最后再发布到社区

### 7.2 这也解释了它为什么能快速扩功能

因为新能力不必重造整套页面，只要接进现有体系：

- 前端入口
- 工具接口
- runtime 消息协议

就能变成一个新 feature。

### 7.3 这也解释了它为什么暂时还是“开环”

虽然它把内容层和工具层拆开了，但目前拆得最深的是：

- 资源接入
- 页面生成
- 页面编辑

不是：

- 学生行为结构化
- 作答数据回流
- 学情反哺 agent

所以它已经是一个成熟的“内容生成与再编辑系统”，但还不是一个成熟的“学习闭环系统”。

---

## 8. 对我们做同类产品的直接启发

### 8.1 资源层一定要先行

不要让每个 agent/tool 自己处理原始文件。应该先有统一的：

- upload
- resourceId
- resourceUrl
- metadata

否则后面所有工具都会高度耦合。

### 8.2 编辑器 runtime 要当成平台，不要当成单页组件

飞象这套最值钱的不是 iframe，而是 iframe 外面那层宿主协议。因为真正可扩展的是：

- 页面请求什么能力
- 宿主能路由什么工具
- 工具结果如何回注页面

### 8.3 文本/图片要做成专项工具，不要全压在主聊天接口

飞象已经证明，局部编辑场景里：

- `ai-generation/text`
- `ai-generation/image`

这种短链路工具调用体验会比“再回到大对话里说一遍”更好。

### 8.4 如果我们要做闭环，下一层必须补 schema

飞象现在的分层已经能很好地解决：

- 资源输入
- 内容生成
- 内容再编辑

但还不能很好解决：

- 题目结构化
- 作答数据结构化
- 学情分析结构化

如果我们要走闭环路线，必须在飞象这套分层之上再加一层：

```text
内容 schema / 题型 schema / 事件 schema
```

否则还是只能做“漂亮的内容机器”，很难做“能回收学习结果的教学系统”。

---

## 9. 一句话总结

飞象的真实技术设计不是“LLM 生成 HTML”这么浅，而是：

> **统一资源层 + 编排 agent + 专项工具接口 + iframe 宿主 runtime + 社区分发层**

其中最值得重视的，不是网页本身，而是**它把外部资源获取、内容生成、页面编辑、内容发布拆成了可以被 agent 调度的一组工具**。这才是飞象能持续长能力、长场景的根本。
