# 飞象（Musk AI Studio）技术路线总结

> ⚠️ **使用须知**：本总结基于一批"对飞象老师 AI 产品交付物（生成的课件 HTML + 一次完整交互日志）的第三方逆向拆解资料"整理而成。
> - 拆解资料分两类可信度：**确证**（有源代码 / URL / 元数据直接证据）与**推测**（从上下文推导、无直接证据）。
> - 飞象**没有公开技术文档**，本文所有"架构层 / workflow 节点 / 知识引擎"等结论，均为从产物反推，**部分为推测，使用需自行判断**。
> - 本文目的：为我们「AI 助教升级」项目提供一个**同类产品的技术参照系**，而非照搬。最重要的一节是文末「对我们项目的可借鉴点与差异点」。

---

## 0. 一句话定位

> **飞象 Musk AI Studio = 一个"对话式输入教学需求 → 多阶段 agent workflow 编排 → 生成自包含可交互 HTML 课件"的 AI 教学内容生成平台。**

它解决的是"AI 把一节课的内容做成可交互的网页课件"这件事，且已经跑通了从"上传复习资料"到"输出 20 页带强互动课件"的完整生产流程。但从拆解资料看，它是一个**开环的内容生成器**——产物是独立 HTML，带浏览埋点（停留时长），**没有看到结构化作答数据回收 / SCORM / xAPI / 学情闭环**。

## 2. 技术栈与平台线索

> 标注：【确证】= 有代码/URL 直接证据；【推测】= 间接线索推导。

### 2.1 平台与域名【确证】

| 域名 / 路径 | 用途 |
|---|---|
| `www.feixianglaoshi.com` | 主站（React SPA 壳 + Frog 埋点 SDK） |
| `musk-online.fbcontent.cn/pub-musk-ai-studio/workflow/file/document/{21位ID}.html` | AI 生成内容的存储与分发 |
| `metis-online.fbcontent.cn/metis-misc/*.js` | 前端库 CDN（命名空间 Metis） |
| `static-nginx-online.fbcontent.cn` | MathJax 额外配置托管 |

- 产品内部代号：**Musk AI Studio**（`pub-musk-ai-studio`）；前端资源 CDN 代号 **Metis**。
- URL 里出现 `workflow` 路径段 → 生成是**工作流/流水线**模式，而非单次 API 调用。
- `watermarkChatId` → 交互是**对话式**（一次生成绑定一个 Chat 会话）。
- `watermarkBizId` → 关联租户/组织，**支持多租户**。
- 外层壳含 `<meta name="generation-source" content="AI生成">` → 飞象**自己明确标注内容为 AI 生成**。

## 3. Agent / Workflow 架构（生成流程怎么编排）

这是飞象拆解里最有价值的部分。一次"上传 71.5KB 九年级物理复习 Word → 输出 20 页课件"的完整交互日志揭示了它**不是单次 LLM 调用，而是 8+ 阶段、含 2 个用户确认门控的 agent workflow**：

```text
① 上传文档        解析 .doc, 提取文本
        │
        ▼
② 任务规划        5 步: 课标研读→教材梳理→学情分析→目标拟定→大纲设计
        │
        ▼
[用户确认门控]    年级/教材版本/课时数/课件页数范围/班级学情/补充需求
        │
        ▼
③ 专家思考        ~2000字: 关键词/课标解读表/知识脉络/物理量清单/
        │              ASCII知识结构树/实验清单/学情误区/学习目标
        ▼
④ 大纲设计        20 页逐页: 页码/类型/教学内容/活动设计/交互方案
        │
        ▼
[用户确认门控]    审阅大纲, 补充需求
        │
        ▼
⑤ 图片生成        文生图: 8 张主题配图 (英文 prompt)
        │
        ▼
⑥ 复杂度评估      分类: 4 页"强互动" + 16 页"普通"
        │              强互动页额外编写"交互剧本"(技术方案)
        ▼
⑦ 分批生成        9 个 Batch 串行生成 + 注入骨架
        │              (强互动页单独占 1 Batch, 普通页 2-4 页合并)
        ▼
⑧ 迭代修复        用户反馈 bug (公式渲染/电路连线) → 针对性修复, 不全量重来
```

**可提炼的编排范式特征：**
1. **复杂度感知路由**：简单单页课件走"单次 LLM 生成"快通道；复杂多页课件走"完整 workflow"。同一系统两套生成策略并存。
2. **Plan → Confirm → Execute**：先规划、用户确认参数、再执行；关键决策点有 **Human-in-the-loop 门控**。
3. **交互剧本中间层**：写代码前先把"教学需求"翻译成"技术方案"（如"Canvas 绘汽缸 + anime.js 控活塞"），是一个独立的设计层。
4. **分批串行生成 + 骨架组装**：长内容不是一次性吐完，而是分 Batch 串行生成后注入统一骨架，每批带前文上下文。
5. **增量修复**：理解"哪段代码要改、哪段不动"，做定向修复（具备增量编辑能力）。

## 4. Skill / 能力体系

飞象的"能力封装"不是传统的**代码组件库**，而是**Prompt 约束规则（Skill）**。拆解的 skill 分析建议一个 Skill 文件应包含：

| Skill 区块 | 内容 |
|---|---|
| 角色定义 | 教学课件生成器，输出自包含 HTML |
| CDN 库声明 | 6 个可用库 + 触发条件（何时用 Three.js / GSAP / anime.js） |
| CSS 变量强制 | 必须定义 `--primary / --background / --text`，可选其余 |
| 交互模式指南 | 按学科知识类型建议交互模式（不强制，AI 判断） |
| 壳框架兼容规则 | 多页输出时交互元素必须用白名单 CSS 类名 |
| 输出格式约束 | 单页（完整 HTML）vs 多页（template + 壳框架）的选择条件 |
| **反噪音约束** | 禁止生成 build marker / GA4 / CMS 时间戳等"生产环境幻觉" |
| 学科正确性 | 公式、数值、单位必须准确 |

**确证的交互模式清单**（从 4 份课件提取，可视为飞象的"题型/互动能力地图"）：

| 交互模式 | 适用场景 | 依赖库 |
|---|---|---|
| 滑块参数调节 | 课件型 | 无 |
| 3D 场景拖拽旋转 | 动画型 | Three.js + OrbitControls |
| 3D 物体动画 | 动画型 | Three.js + GSAP/anime.js |
| 点击填空检测 | 游戏型 | 纯 DOM |
| 拖拽排序匹配 | 练习型 | HTML5 DnD |
| Canvas 绘图白板 | 讲解型 | Canvas 2D |
| 逐步揭示内容 | 讲解型 | anime.js / CSS |
| 双人竞技答题 | 游戏型 | anime.js |
| 填空输入 + 错因弹窗 | 练习型 | contenteditable |
| 关卡闯关 | 游戏型 | 状态机 |
| 图片热区覆盖 | 通用 | CSS 定位 |
| 视频嵌入 | 引入型 | video 标签 |

> **对我们的关键启示（提前剧透）**：这套"交互模式"在飞象里是**每次重新手写**的、无统一 schema 的。它能"演示互动"，但**无法标准化采集每一道题的作答数据**——这正是飞象（开环）与我们（要闭环采集）的根本分水岭，详见 §7。

## 5. 前端与内容运行时

### 5.1 两种交付形态【确证】
- **单页课件**：一个完整 HTML 文档塞进 `iframe srcdoc`，自包含、可独立访问。
- **多页课件**：壳框架 + 多个 `<template class="page-data">` 分页 + 翻页脚本（手势/键盘/缩略图/演示模式）。

### 5.2 壳框架的事件拦截机制【确证】
壳框架拦截页面空白处点击用于翻页，但用**约定 CSS 类名白名单**自动放行可交互元素（`.tool-btn / .drag-item / .drop-zone / .option-btn / .input-box / canvas / video ...`），并用 `MutationObserver` 监听动态元素。
→ AI 生成的课件只要**遵守类名约定**，就能被壳框架正确识别。这是"约定优于配置"的轻量运行时，**不是 schema 驱动的渲染引擎**。

### 5.3 内容载体【确证】
- 纯代码生成（CSS 手绘 + JS 逻辑），或
- **图片 + 代码混合**：用预制设计图（jpg/png，来自 Musk CDN `user/upload/`）作底图，叠加交互代码。图片可能来自设计师或文生图模型。

### 5.4 内容结构（无 schema/DSL）【确证】
飞象**没有内容数据结构（schema/DSL）**这一层。内容就是 HTML 代码本身。三份课件之间零代码复用，CSS 命名 / JS 组织（OOP/函数式/全局状态）/ DOM 设计完全各异。
→ 优点：表达力极强、不受 schema 束缚；缺点：**不可结构化采集、不可版本化演进、不可标准化质检**。

## 6. 数据与闭环

**这是飞象拆解里最"空"的一节，恰恰是我们最强的一节。**

- 飞象产物只挂了 **Frog SDK 埋点**，确证能力仅到"**页面停留时长统计**"级别。
- 拆解资料中**没有任何 SCORM / xAPI / 作答数据回收 / 学情沉淀 / 数据反哺**的证据。
- 它的"知识"是**单向供给**：教育知识引擎（课标/教材/学情模型，推测）在**生成时**注入上下文，但**没有看到学生作答数据回流去更新知识或反哺下一次生成**。

**结论：飞象是"开环生成"**——生成完即交付，学生怎么作答、答对答错、薄弱在哪，系统拿不到也不消费。

## 7. 对我们「AI 助教升级」项目的可借鉴点与差异点 ★最重要

> 我们的项目目标：让 ClassIn AI 助教从 RAG 问答升级为"替教师完成教学动作"的 agentic 入口，第一条链路是「互动内容生成 · 多场景交付 · SCORM 数据回收 · 反哺生成」。

### 7.1 可以直接借鉴（飞象已验证可行的）

| 借鉴点 | 飞象做法 | 对我们的意义 |
|---|---|---|
| **多阶段 agent workflow** | 8+ 阶段 + 2 个用户确认门控，而非单次 LLM 调用 | 验证了我们"生成=agent/workflow"的判断方向正确，可直接参考其阶段划分 |
| **复杂度感知路由** | 简单课件走单次生成、复杂课件走完整 workflow | 我们可用同样思路控制成本/延迟：简单题型快通道，复杂内容走全流程 |
| **Plan→Confirm→Execute + HITL 门控** | 规划后停下来让老师确认参数（页数/学情/重点） | 教师对生成有掌控感、降低废稿率，值得照搬到我们的生成交互里 |
| **交互剧本中间层** | 写代码前先把教学需求翻成技术方案 | 对应我们的"规划→结构化生成"之间可加一层方案设计，提升可控性 |
| **教育知识引擎做生成上下文** | 课标/教材/学情（推测）注入生成 | 印证我们「重点5·知识库对接」：知识库是生成的认知底座 |
| **CDN 库白名单 + 反噪音约束** | 限定可用前端库、禁止幻觉式生产代码 | 我们生成结构化内容时同样要约束输出范围、做净化 |
| **分批生成 + 骨架组装** | 长内容分 Batch 串行注入骨架 | 我们做长篇/多题内容时的稳妥工程方案 |
| **增量修复** | 老师反馈 bug 后只改目标段 | 对应我们"教师可编辑修正"，避免全量重生成 |
| **能力即 Skill（约束规则封装）** | 把"用什么库/什么结构/什么约束"封装成 Skill | 我们可把"题型生成规则"封装成可复用 skill，但要叠加 schema（见差异） |

### 7.2 不适用 / 必须做得不一样（我们的差异化）

| 维度 | 飞象（参照系） | 我们必须不同 | 为什么 |
|---|---|---|---|
| **闭环 vs 开环** ★ | 开环：只埋点停留时长，无作答回收 | **必须闭环**：SCORM/xAPI 采集作答 → 学情沉淀 → 反哺生成 | 这是 ClassIn 的护城河，是项目核心价值 |
| **内容结构** ★ | 无 schema，每次生成整页 HTML，零复用 | **必须 schema/DSL 驱动**：题型组件库 + 内容运行时 player | 只有结构化内容才能标准化**采集作答数据**；飞象的"整页 HTML"路线天然无法采集 |
| **运行时** | 翻页壳框架（约定类名放行点击） | **schema 驱动的渲染引擎 + 行为埋点**，与 SCORM 运行时衔接 | 我们要采集"哪题、答案、对错、用时"，需要 player 统一接管交互与采集 |
| **生成产物** | 独立 HTML，浏览即止 | **可投放进 ClassIn 预习/课堂/作业/表单**，对接 LMS 活动体系 | 飞象不与 LMS/教室/班级数据集成，我们要原生进场景 |
| **知识流向** | 单向：知识→生成 | **双向**：知识库供给生成 + 作答数据回流更新学情/反哺 | 飞象没有"记忆系统"，我们要把数据回路闭上 |
| **质量保障** | 无后处理、噪音原样保留 | **要校验/重生成 + 人工可修正** | 教学内容正确性要求高，且作答数据正确性不能错 |

### 7.3 三条最关键判断（给 PM 和研发的结论）

1. **架构分水岭就在"内容结构"**：飞象证明了"整页 HTML 一次性生成"在**纯展示型课件**上完全可行且表达力强；但它**无法支撑结构化作答采集**。我们一旦要做数据闭环，就**不能走飞象的无 schema 路线**，必须在 Phase 0 就把"内容 schema/DSL + 题型组件 + 运行时 player"这条地基定下来。这是我们和飞象最根本的技术分歧，也是选型时最该守住的一点。

2. **飞象的 workflow 编排可以"抄作业"，但闭环要自己造**：生成侧的多阶段编排、复杂度路由、HITL 门控、交互剧本、增量修复，飞象都验证过了，我们可以大胆参考、少走弯路；但 SCORM/xAPI 采集 → 学情沉淀 → 反哺生成这整条**数据回路飞象根本没有**，是我们必须从零设计、也是真正拉开差距的地方。

3. **我们的差异化叙事成立且有据**：飞象（及 musk-ai-studio）代表了"AI 生成互动课件"这一开环赛道的较高水位——他们做得相当完整（多模态、20 页复杂课件、专家级教研逻辑）。但他们止步于"生成 + 交付独立 HTML"。**ClassIn 手里的在线教室、LMS、SCORM、班级数据，是飞象拿不到的闭环要素**。我们的价值不在"比飞象生成得更快/更好看"，而在"把生成的内容**投进真实教学场景并把数据收回来反哺**"——这是别人开环、我们闭环的结构性优势。
