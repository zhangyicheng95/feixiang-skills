### 1.4 关键架构结论
飞象**没有"题型组件库 / 内容 schema / 播放器运行时"**。每个课件的 CSS 类名、DOM、JS 组织都是 LLM 从零生成、互不复用。它的"运行时"只是一个**翻页壳框架**，靠**约定 CSS 类名白名单**识别可交互元素放行点击，**不是 schema 驱动的渲染引擎**（来源 A/B，跨样本验证见 §5）。

## 2. Agent / Workflow 编排逻辑（生成全流程阶段拆解）

### 2.1 端到端流程（合并 A 的交互日志 + B 的产物流水线 + C 的前台 UI + E 的 runner 观察）

```text
①意图理解&教学设计 → ②大纲/分页规划(outline) → [用户确认门控①]
   → ③专家思考(~2000字) → ④逐页大纲设计 → [用户确认门控②]
   → ⑤图片生成(文生图) → ⑥复杂度评估(强互动 vs 普通)
   → ⑦分批串行生成(9 Batch + 注入骨架) → ⑧素材回填 → ⑨互动/状态装配
   → ⑩后处理(噪声/水印) → ⑪外层壳封装(iframe + Frog埋点) → 交付物卡片
   → [追问] → ⑫增量定向修复 → 新版交付物卡片
```

实测案例（来源 A）：上传 71.5KB 九年级物理复习 Word → 输出 20 页课件，全程 8+ 阶段、含 2 个用户确认门控、9 个 Batch 串行生成。

### 2.2 前台 UI 对应的可视化阶段【实证，来源 C/E】
信息确认表单（AI 预填默认值，字段带"刻"后缀术语）→「已确认」标签 → 专家思考可折叠卡片（关键词/课标研读/教材检索/精选资源/目标重难点/逐页设计表）→ 课件生成计划（强互动页 + 普通页 + 9 Batch）→ 分批生成进度行（🔄→✓ 逐条）→ 交付物卡片（HTML 图标 + 缩略图 + 下载）→ AI 说明文字 → 追问修复闭环。

### 2.3 可提炼的编排范式特征（来源 A/B）
1. **复杂度感知路由**：简单单页课件走"单次 LLM 生成"快通道；复杂多页课件走"完整 workflow"。两套策略并存。
2. **Plan → Confirm → Execute**：先规划、用户确认参数（年级/教材版本/课时/页数/学情/补充需求），再执行；关键决策点有 **HITL 门控**。
3. **交互剧本中间层**：写代码前先把"教学需求"翻译成"技术方案"（如"Canvas 绘汽缸 + anime.js 控活塞"），强互动页额外编写"交互剧本"。
4. **分批串行生成 + 骨架组装**：长内容分 Batch 串行生成后注入统一骨架；强互动页单独占 1 Batch，普通页 2–4 页合并。
5. **增量定向修复**：用户反馈 bug（如公式渲染）→ 只改目标段，不全量重来。

### 2.4 教学设计的成品结构【实证，来源 B】
20 页课件呈"总—分—练—结"四段式 + 双模块对称（热学 3–8 / 电学 10–17）+ 转场页（9）做模块切换锚点。每页带教研语义标签 `data-name`（封面/复习框架/热机四冲程/串并联电路连线…）。强互动是**稀缺投放**（20 页仅 3–4 页重互动），由"知识点是否含动态过程"触发。

### 2.5 流式机制【实证，来源 E】
agent runner 用 **HTTP 高频短轮询** `/musk-ai-studio/api/agent-chat/executing-status`（约 80–150ms 一次）拉执行状态，**而非 SSE/WebSocket**，以适配长耗时批量生成；文字**整块出现**，非逐字流式。

### 2.6 能力体系 = Skill（Prompt 约束规则，非代码组件库）【来源 A】
一份 Skill 文件应含：角色定义、CDN 库声明 + 触发条件、CSS 变量强制、交互模式指南、壳框架兼容规则（白名单类名）、单页/多页输出格式约束、**反噪音约束**（禁止 AI 幻觉生成 build marker/GA4/CMS 等）、学科正确性约束。
> 注意方向相反：**生成阶段反噪音**（Skill 禁止幻觉元数据），**发布阶段加噪音**（后处理注入垃圾，见 §3.4）。

确证的交互模式清单（飞象"题型/互动能力地图"，来源 A/B）：单选/判断、分步揭示、3D 拖拽旋转、anime 动态演示、原生 DnD 建模、DnD 归类、Canvas 白板、点击填空、双人竞技、关卡闯关、图片热区、视频嵌入。

## 3. 课件产物运行时与内容数据结构

### 3.1 两种交付形态【实证，来源 A/B】
- **单页课件**：一个完整 HTML 塞进 `iframe srcdoc`，自包含。
- **多页课件**：`page-shared`（设计令牌+共享库）+ 20 个 `<template class="page-data" data-id data-name>` 分页 + 壳框架翻页脚本（手势/键盘/缩略图/演示模式）。

### 3.2 页骨架三件套【实证，来源 B，courseware 4561–5485 行】
- 分片容器 `<template class="page-data" data-id data-name>`：每页一个，`data-id` 连续。
- 页容器 `.page-container`（18/20 页用；封面/转场/结课为全屏特例页）。
- 组件基线类：`.page-title`（32px/700/左 6px 主色竖线）、`.card`（圆角 12 + 阴影）、`.btn` / `.btn-primary`。
- 栅格几乎全靠**行内 style 写 flex/grid**，未抽象出栅格类 → outline 只约束"页级骨架"，不约束"块级布局"。

### 3.3 设计令牌【实证，来源 B/D】
唯一 `<template class="page-shared">` 内一处 `:root` 定义 7 变量：`--primary / --secondary / --accent / --bg / --text / --success / --error`。但**状态反馈色大量写死 hex**（如正确 `#dcfce7/#22c55e`、错误 `#fee2e2/#ef4444`），未复用 `--success/--error` → 令牌是**提示词级软约束**，编译期不强制，AI 经常旁路。

### 3.4 内容数据结构 = 无 schema / 无 DSL【实证，来源 B，★关键】
内容就是 HTML 代码本身，没有"题目对象/选项数组/答案字段"这一层：
- 单选题：`checkQ7(this, true/false)`，**正确答案以布尔字面量硬编码在 `onclick`**，题干/选项/解析全是 DOM 文本。
- 分步揭示：`showStep()` 靠计数器自增。
- 动态演示：`runStroke(index)` 用数据数组驱动绝对定位 DIV + anime（**非真 `<canvas>`**，搜 `getContext` 0 命中；电路页 `<svg>` 留空未用）。
- 拖拽建模：`testCircuit()` 仅查"元件是否齐全"，**不验真实连线拓扑**（AI 自述注释"简化版…不包含复杂的实时SVG拓扑"）。
- 拖拽归类：`handleDrop()` 靠 `data-type` 字符串相等匹配。
- 命名约定：函数=动词+对象（check/show/run/play/test），id=语义-序号（step-1/zone-series）。
- **零组件抽象**：同名 `saveState()`/`state` 在多页各写一份，靠 `<template>` 隔离作用域 → "每页一份拷贝"，非模块化。

→ 后果：表达力极强但**不可结构化采集、不可版本化演进、不可标准化质检**。

### 3.5 状态协议 = postMessage（有名无实）【实证，来源 B】
课件内 `window.parent.postMessage({type:'saveState', state}, '*')` 上抛，但 **state 形状逐页自定**（`{mode}` / `{currentStroke}` / `{items}` / `{correctCount}`），无统一 schema，无法跨页聚合；恢复能力参差（部分页"（略）"放弃、部分页根本无监听）。**且 parent 根本不消费 saveState** → 作答数据在 iframe 边界断流（详见 §4）。

### 3.6 素材 / 资源管线【实证，来源 B/D】
- 配图：文生图 → 对象存储（22 位随机 ID）→ 回填 `<img src>`，与课件 HTML 同一 `workflow/file/` 目录。20 页仅 4 张图 → "**能代码画的就代码画，画不出的真实照片才文生图**"的节流规范。
- 共享库走 CDN 白名单（`metis-misc/*.js`），课件只引用不内联打包；多样本复用同一组库文件【实证 D，见 §5】。
- CSP：仅约束 `img-src` / `media-src`（自有 fbcontent + 阿里云 OSS + GCS），**不约束 script/style** → 噪声里的假外链不会真请求。

### 3.7 SCORM / iframe / 埋点【实证，来源 B/C】
- **iframe**：外层壳 `iframe srcdoc`（HTML 实体编码）+ `__SERVER_DATA__` 注入 4 个 watermark 变量 + `file://` 自动重定向到在线地址。
- **埋点（Frog SDK）只收浏览行为**：`use_time`（停留秒数，多事件 + 100ms 防抖累加，很讲究）、点 logo、点"立即体验"；全页点击事件被注释未启用。
- **SCORM / xAPI：完全没有**【实证：资料中零证据，来源 A/C】。课件不符合任何 e-Learning 标准，无法被 LMS 追踪。课件标称 960×540（C）/ 固定 px 画布（B），自适应靠外层壳等比缩放（移动端 375 rem 基准）。

## 4. 数据与闭环能力（飞象能/不能回收什么）

| 能力 | 飞象现状 | 证据 |
|---|---|---|
| 页面停留时长 | ✅ 能（Frog `use_time`） | 【实证 B/C】 |
| 品牌点击（logo / 立即体验导流官网） | ✅ 能 | 【实证 B】 |
| 学生作答（题/答案/对错/用时/尝试次数） | ❌ 不能 | saveState 上抛但 parent 不消费，iframe 边界断流【实证 B】 |
| SCORM / xAPI 标准回收 | ❌ 无 | 资料零证据【实证 A/C】 |
| 学情沉淀 / 数据反哺生成 | ❌ 无 | 知识单向供给：知识→生成，无回流【推断 A】 |
| 与 LMS / 班级 / 课堂数据集成 | ❌ 无 | 生成与使用是两个孤岛【来源 C】 |

**结论：飞象是"开环生成"**——生成完即交付独立 HTML，学生怎么作答系统拿不到也不消费。埋点投入全压在"停留时长"这一个指标上，恰恰反衬作答数据完全未采。这是飞象与"需做闭环采集"产品的根本分水岭（来源 A/B/C）。

## 5. 跨样本验证结论（软规范 vs 硬资产）

本次用 Grep 抽样 4 个课件样本（courseware / s2 / s3 / s4 的 `.inner.html`）交叉验证（来源 D + E）：

### 5.1 被工程化的"硬资产"（跨样本一致）【实证 D】
- **共享前端库**：`metis-misc/zgLDUdmazTYc0B4K6Cor.js`、`zZZY40t7WJC7UdQCPACm.js`、`blER0Bn7vsa2JER9IEssf8.js` 等同一组文件在多个样本中复用 → 库层是固定白名单资产。
- **外层壳 + iframe 宿主 + Frog 埋点**：结构跨样本一致。
- **后处理噪声层**：`WCAG 2.1 AA`、`Webpack 5.88.2`、`Marker-xxxx`（`left:-9999px` 离屏隐写水印）、`data-measurement-id`(ga4)、`feature-flags`、统一 CSP 等假元数据，在 s4 等样本中同样成体系出现 → 发布后处理流水线是工程化的、可批量注入的。

### 5.2 靠约定的"软规范"（跨样本各异）【实证 D】
- **设计令牌值逐课件不同**：`--primary` 在四个样本分别是 `#3B82F6`(courseware) / `#2563EB`(s4) / `#01553D`(s3) / `#D7F324`(s2)，`--accent` 也各异。**令牌名一致、取值由 LLM 现场决定**。
- **内层 DOM/CSS/JS 零代码复用**：每份课件结构各写各的，互动函数手写、命名约定相同但实现不共享。

### 5.3 核心判断【来源 E，本次实证强化】
> **内嵌设计系统是"软规范"**（靠 prompt/示例约束，而非硬资产）：**只有外壳被工程化**（埋点、iframe 宿主、共享库、噪声/水印），**内层内容由 LLM 生成、风格一致性靠约定维持**。换言之，飞象的"设计系统"不是一套可复用的组件/主题代码，而是一组写进 Skill 的命名与令牌约定——所以跨样本能看到"名字一样、值不一样、代码不复用"。
