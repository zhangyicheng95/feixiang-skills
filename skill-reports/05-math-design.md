## 1. 设计思路

### 1.1 问题定位：不是「不会配色」，而是「配色塌缩」
SKILL.md 第 17 行直接写明动机来源——石墨测试《数学案例·色彩skill应用效果测试（4.22）》的三个量化结论：

- **skill 命中率约 50%**（54 案例中只有约一半真正启用了 skill）；
- **命中后 25 套色板仅用到 6 种，且高度重复 A-01**（多样性塌缩）；
- **15/25 命中案例 H1 字号错误**（规范被忽略）。

整套 SKILL 的设计，本质是针对这三个失败指标的**反制工程**，而非泛泛的「设计美学指南」。这是它最值得学习的地方：**skill 不是写规范，而是写"对抗模型坏习惯的强制流程"**。

### 1.2 三层硬约束体系
- **视觉身份（§一）**：16:7 可视区、一屏优先、按钮统一 80px、安全区、**禁止文字前 Emoji，元素一律 SVG 几何绘制**（平涂、无阴影、块面感）。
- **布局（§二）**：给出可执行的布局公式 `Container(flex)=Header(fixed)+Stage(flex-grow:1,overflow-y:auto)+Controls(flex-shrink:0)`，确保标题与按钮常驻首屏，仅中部演示区局部滚动。
- **字号（§三）**：H1=40px/700!important、H2=30px/600、H3=28px、Body=28px、Caption≥22px、按钮 28px。`!important` 是为了「防止被框架覆盖」——这是被线上坏例倒逼出来的防御写法。

### 1.3 学段分色心智模型
- 小学 1-6 → **pool A 活力高饱和**（11 套）；初高中 7-12 / 代数·函数·三角·数列 → **pool B 智性低饱和**（20 套）。
- 色板不是单色，而是**完整令牌组**：背景色、文字色、主色调、卡片(background/border/radius)、核心按钮、次要按钮、**演示区背景+网格(linear-gradient 点阵)**、**知识演示区 3D/几何 100% 色**、设计关键词标签。这让「数形结合演示区」具备工业级一致性，而非随手取色。

## 2. 迭代路线

### 2.1 可追溯的迭代痕迹（同一文档内部的"地层"）
本 skill 没有 CHANGELOG，但文档内部存在**自相矛盾的"地质分层"**，正是迭代留下的化石：

1. **H1 字号的 40 vs 42 之争**：
   - §二（布局）第 67 行：「默认 H1 为 42px……优先通过 `clamp(30px,5vh,42px)` 缩小」。
   - §三（字号）第 75 行 + Phase 5：「H1 必须 40px……**禁止用 42px 或其他值覆盖第三节 H1 规范**」。
   - 这是**先有 42px 旧规范，后因石墨测试新增 40px 强约束**，但旧段落没删干净。Phase 5 的"禁止 42px"实际上是在和本文档自己的 §二 打架。

2. **色板文件头部的"全量读" vs SKILL 的"按需读"**：
   - `color-palettes-A_活力高饱和.md` 第 2-6 行：「**必须读取所有色彩方案，包含 A 01-11 和 B 01-20 共 31 种**……禁止固定第一个」。
   - SKILL.md Phase 4：「**禁止全量加载 31 套**，pool=A 只读对应编号整段」。
   - 即从「全量加载 + 随机」演进到「按需加载单段 + 随机」——这是 **token 成本优化迭代**（31 套全读会爆上下文），但色板文件的旧头部没同步更新。

3. **`v1/visual-impact.md` 与根目录 `visual-impact.md` 完全一致**，说明 visual-impact 在 v1 后未再改动，是稳定内核。

### 2.2 数据质量缺陷（生产数据的真实噪声）
作为「真实生产级资料」，色板文件带着大量未清洗的脏数据，恰恰是它真实的佐证：
- **编号错乱**：B 系出现 `#### 色彩04`（漏 B 前缀）、`#### 色彩色彩B-15`（重复"色彩"）。
- **重复定义**：`B-10` 与 `B-11` 令牌**完全相同**；`A-09` 与 `B-19`、`A-03` 与 `B-13`、`A-04` 与 `B-17`、`A-08` 与 `B-14` 等**跨池重复**（A 系部分色板被整体复制进 B 系）。
- **非法色值**：A-05 网格 `#F7D61A9`（7 位 hex 非法）；A-08 文字色 `#F`（截断）。
- **游离字符**：A-10 段末突兀的一行 `B-`。

> 这些缺陷不影响"随机选 1 套整段读取"的运行（模型读到哪段用哪段），但说明色板库是**人工教研逐步堆叠、缺乏 schema 校验**的产物。

## 3. 功能边界

**纳入（必启用）**：数学 + (教学动画/互动课件/HTML/三维/闯关/算理/几何/方程/函数/统计/概率)，或 prompt 含「互动设计/习题配置/三维展示/呈现形式」批量测试结构。触发词覆盖极细（勾股定理、鸡兔同笼、垂径定理、诱导公式、等差数列、概率转盘……），是为提高"命中率"刻意堆叠的关键词诱饵。

**排除（不启用）**：多页 PPT 课件（→ `courseware-generator`）、纯文字无 HTML 答疑、非数学学科。

**与邻居 skill 的关系（关键）**：
- **被 html-authoring 收编**：html-authoring v5 第 31 行规定数学场景"必读 math-design/workflow.md → 色板子文件 → visual-impact.md"，并设了严格的"学科路由第 0 步"：**非数学学科即使含"教学动画/互动课件"也禁止读 math-design**。本目录正是这套子链路的色板/字号核心。
- **与 courseware-generator 互斥**：courseware-generator 处理 960×540 多页 PPT；math-design 显式排除多页场景。两者构成「单页 vs 多页」的分工。
- **与 physics-formula-typography 正交叠加**：数学/物理单页 HTML 中，math-design 管"配色+布局+字号"，physics-formula-typography 管"公式排版"，二者在同一产物上叠加生效。
- **被 test-html 验收**：math-design 的硬约束（H1 字号、按钮高度、单屏无溢出、调色板白名单）正是 test-html「约束符合性（模板 6）+ 响应式（模板 2）」断言要扫的对象。

## 4. 工程启发（对 ClassIn 互动内容生成项目）

1. **用"失败指标"反向驱动 skill 设计**：math-design 最大的方法论价值不在色板本身，而在它把石墨测试的三个量化失败（命中率 50%、色板塌缩到 A-01、字号错 15/25）转成 Phase 1-5 的强制动作。ClassIn 互动内容生成应**先建立基线评测（命中率/多样性/规范符合率），再以指标缺口写约束**，而不是凭直觉堆规范。

2. **对抗"默认塌缩"必须显式注入随机**：模型天然偏好第一个选项（A-01）。Phase 3「⚠️ 禁止固定 A-01/B-01，均匀随机 1 个编号，过程输出标注 palette_id」是廉价而有效的反塌缩手段。ClassIn 若有模板/主题/配色池，应同样**强制随机 + 可观测标注**（输出 `palette_id` 便于事后审计多样性）。

3. **按需加载替代全量加载**：从"读 31 套"到"只读选中编号整段"是直接的 token/成本优化。ClassIn 的设计令牌库应做成**可寻址分段**（`#### A-XX` 锚点），SKILL 只声明"读哪一段"，把大资产挡在上下文之外。

4. **设计令牌要给"演示区/3D"单独成组**：色板里专门有"知识演示区 100% 色（避免光照过暗）"令牌组，区别于卡片/按钮色。ClassIn 的几何/3D/图表演示区配色应**与 UI 配色解耦**，否则演示对象会被 UI 中性色拖暗。

5. **规范要写"防覆盖"细节**：`H1 ... !important`、按钮固定 80px、安全区 20px——这些都是"被框架/模型覆盖"后补上的防御。启示：**硬规范要落到可机检的具体数值并加防御性写法**，模糊的"字号要大一点"必被忽略。

6. **反面教训：缺 schema 校验会沉淀脏数据**。色板库的编号重复、非法 hex、跨池复制，提醒 ClassIn 的资产库（主题/模板/题型）从第一天就要有**结构校验 + 唯一性约束 + lint**，否则人工堆叠会持续累积静默错误，且随版本被复制扩散。

7. **同文档矛盾是迭代失控信号**：40 vs 42 的字号打架说明"在旧文档上打补丁"易留矛盾。ClassIn 的 skill/规范应有**版本化 + diff 审查**，新增约束时强制清理被取代的旧条款。
