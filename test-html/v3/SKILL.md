# test_html 使用指南

> 本文是「纪律与决策」主干。**详细代码模板、致命写法清单、完整示例、反面教材**都在
> `references/test-templates.md`——动手写 `playwrightCode` 时按需 `read_url` 该附件取用，不要凭记忆默写。

## 何时触发

1. **生成/编辑后验证**：`create_file` / `edit_file` 改完 HTML（多次编辑全部完成后再测），验证功能与布局。
2. **用户反馈诊断**：反馈功能异常（点不动/没反应）、布局问题（手机乱/截断/横向滚动条）、渲染异常（公式/图片/重叠）时，**先写复现测试确认问题，再修**（先测后改）。
   - 不适合测的（直接读代码分析）：视觉审美（"不好看"）、性能感受（"卡/慢"）。

## 前置条件

- 需要明确 `resourceId`（来自 create_file/edit_file 返回或用户提供）；不能同轮生成文件又测试，要等拿到 id。
- 代码格式要求（顶层语句、可用 `page/context/browser/expect` 免 import、等待与诊断方式）见 `test_html` 工具 `playwrightCode` 参数说明。
- **写代码前先读 `references` 顶部「5 条致命写法」**（JS 正则字面量 / f-string 拼 JS / strict mode 多匹配 / range `fill()` / 用 `re·json` 须 import）——这几条是"一行没跑就 SyntaxError"级，凭记忆默写极易踩。
- **单次云函数最长 60s**：等待统一 `page.wait_for_timeout(ms)`，单条 ≤2000ms，**总等待 ≤20s**；复杂场景拆成多次调用，每次只测一个维度。

## must-cover 清单（测试的灵魂）

测试要针对当前任务的真实需求，不是机械套模板。依据本次会话**所有用户消息**（含后续每次"再改一下"）抽清单，
不要只看最近一条。清单有四类来源，都要覆盖：

| 来源 | 怎么转成断言 |
|---|---|
| **① 动词+名词（交互）** | 触发动作 + 触发前后状态比对（三步法：名词→存在/计数；动词→`mouse/keyboard/click`；状态→等信号再断言） |
| **② 声明式约束**（禁止 / 只准 / 全部 / 每个 / 不得 / 固定为 / 至少 N 个） | **否定式或全称断言，全局扫，不是单点**——"<u>这个</u>背景色对"要写成"<u>全页</u>无禁用色"，"<u>有</u>水印"写成"<u>每页都有</u>水印" |
| **③ 核心玩法闭环**（游戏/答题/模拟器/互动演示） | **清单第一条**：端到端走通并产生反馈（如 24 点：选数字→选运算符→判定=24→对错反馈→重置），**而非只验"按钮存在"** |
| **④ 交付可达性**（多文件 / index 大厅 / 有跳转链接 / 已发布资源） | **逐链接 `page.goto` 断言不是 404/NoSuchKey、且目标核心元素渲染**——"链接 `<a>` 存在"≠"目标能打开" |

**品类常漏项补盲表**（"用户一笔带过却常漏"的隐性项；看到对应品类主动对照查漏，用户没要求的不硬塞）：

| 品类 | 常漏项（对照补进 must-cover） |
|---|---|
| 教学游戏 / 答题闯关 | 答对反馈 · 答错反馈（"想一想/再试试"） · 重置 · 计分 · 下一题/难度递进 |
| 拖拽放置 | 拖拽真生效（DOM 变化） · 错位回弹/拒绝 · 命中判定 · 数量上下限 |
| 单页动画 / 演示 | 进度滑块/步骤切换 · 自动播完信号 · 可重置 |
| 课件单页 | 960×540 主画布无溢出 · 主交互可触发 · 375px 不破版 |
| 错题诊断 / 多步状态机 | 上传/输入触发渲染 · 各状态切换 · 兜底文案 |

**格式**：每段 `playwrightCode` 顶部先写 `# must-cover:` 清单，每条 `[N]` 对应一处断言；
某条不测就在其下写 `# skipped: <一行理由>`——**不要默默省略**。

**spec 优先播种**：产物首行有 `<!-- spec: ... -->`、或 `task_write.spec`/`spec_ref` 有契约时，
据它逐条播种清单（`forbid / require / core-loop / count / links`），不另起炉灶；spec 外置为资源（`spec_ref=resourceId`）时先 `read_file` 再转断言。

> **硬门（这两条是红线）**：来源③核心闭环、来源④可达性**未通过 → 不得判定交付完成、不得 terminate**。
> 这两类正是线上反复翻车点（交付"连运算符都没有"的游戏外壳 / 点开链接 NoSuchKey），靠"按钮存在/能渲染"的表层断言兜不住。

## 怎么写断言（模板速查）

三步法选断言后，按产物特征到 `references/test-templates.md` 取对应模板代码：

| 产物特征 | 取哪个模板（详见 references） |
|---|---|
| 任何 HTML | 基础渲染（必测） |
| CSS 布局 / flex / grid | 响应式（375/768/1280，查水平滚动条 + 元素溢出） |
| MathJax / KaTeX / `\(` / `$$` | LaTeX 渲染（引擎识别 + 错误 + 零尺寸） |
| 拖拽 / 键盘 / 动画 | 复杂交互（真实拖拽用 `mouse.*` 非 `drag_to`；动画等信号不硬等） |
| 业务逻辑暴露在 `window` | `evaluate` 调底层 API（最稳，验逻辑而非 UI） |
| **禁止 / 只准 / 全部 / 每个 / 固定为** | **约束符合性**（否定式 + 全称，`getComputedStyle` 全局扫） |
| **游戏 / 答题 / 模拟器** | **核心闭环端到端**（must-cover 第一条） |
| **多文件 / index 大厅 / 有链接** | **资源可达性**（逐链接 `goto` 验不 404 + 渲染） |
| img / video / audio | 资源加载（`naturalWidth` / `complete`） |

按特征把多个片段组合进**一次** `test_html` 调用，减少调用次数；逼近 20s 预算时再拆。

## 失败处理（判定流程）

**默认假设 HTML 有问题。** 除非满足以下任一，**否则不准把矛头指向测试代码**：
(a) Playwright API/语法错（不存在的方法、拼写错）；(b) **已用诊断模式打印过实际 DOM**、证实渲染对只是断言写错；(c) NameError/SyntaxError。

```
失败 → 是 (a) API/语法错？ 是→直接修测试
      否→跑[诊断版]（只打印不断言；用 page.evaluate("console.log(...)") 灌 page console，Python print 不回传）
          ├ DOM 无元素 / state 空 / 数值差巨大 → 改 HTML（edit_file），测试不动
          ├ 渲染对、值合理，只是断言写错 → 用诊断到的实际值收紧断言（绝不无理由放宽）
          └ 超时（timeout / Read timed out / 云函数失败）
              ├ console 有页面级错/死循环 → HTML 死循环或同步重算：拆任务/节流/异步化
              └ console 正常、测试自身在等 → 测试太慢：拆调用/削视口循环/缩 wait
```

- **同一段代码连续失败 ≥2 次** → 强制切诊断版（至少把上次失败的实际值写进断言）。
- **严禁逐轮提交字节级相同的 `playwrightCode`**（只改 HTML 不改测试 = 同一把尺子量同一段错码，结果不会变）——每轮至少换断言 / 选择器 / 粒度其一。
- **同一思路连续失败 ≥3 次** → 换修复策略（换布局 flex↔grid / 换实现 / 换选择器 / 查缺失依赖）。
- **禁止**：无理由放宽断言、删除或跳过失败用例、连续失败直接放弃。

## 收尾产出格式（必填，纯中文，给用户看）

每次拿到结果后，回复里必须输出结论卡（不要用"测好了已修复"含糊带过，不要出现内部工具名/英文术语）：

```
## 验证结论
- 状态：✓ 完全通过 / ⚠ 部分通过 / ✗ 未通过
- 需求覆盖：N/M 项已验证
  - [1] <清单条目>：✓ 已验证 <断言点> / ✗ 未覆盖 <原因>
- 主要原因（⚠/✗ 时）：页面实现问题 / 验证代码问题 / 执行超时
- 下一步：修复页面 / 调整验证代码 / 拆分后再验 / 完成交付
- 仍需补测：<未覆盖清单项；全覆盖填"无">
```

- **只有「状态=✓ 完全通过 且 N/M 全中」才算可交付**；通过 ≠ 没问题——`pass=true` 只代表"写出来的断言满足"，
  标记完成前回看顶部 must-cover：任一 `[N]` 在断言区找不到对应代码 → 仅"部分通过"，补测后再推进。
- **测试设施异常**：连续 ≥3 次"云函数失败/基础设施错"（非断言失败、console 空）→ 停止重试，收尾**仍出卡片**
  （状态 ✗、主因"执行超时/设施异常"），明确告知用户**未经自动化验证、建议人工检查核心交互**；**禁止假装通过、禁止只字不提**。
